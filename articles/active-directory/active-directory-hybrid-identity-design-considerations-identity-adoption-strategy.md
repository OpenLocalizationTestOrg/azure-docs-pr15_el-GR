## <a name="define-business-needs-strategy"></a>Ορίστε τη στρατηγική επιχειρηματικές ανάγκες
Η πρώτη διευθύνσεις εργασίας Καθορισμός επιχείρηση εταιρείες ανάγκες.  Αυτό μπορεί να είναι πολύ μεγάλη και ολίσθησης εμβέλειας μπορεί να προκύψει εάν δεν είστε προσεκτικοί.  Στην αρχή Διατηρήστε το απλό αλλά να θυμάστε πάντα για το σχεδιασμό για μια σχεδίαση που θα περιλαμβάνει και να διευκολυνθεί η αλλαγή στο μέλλον.  Ανεξάρτητα από το εάν είναι μια απλή σχεδίαση ή ένα πολύ σύνθετη, Azure Active Directory είναι η πλατφόρμα Microsoft ταυτότητας που υποστηρίζει το γνωρίζετε εφαρμογές του Office 365, υπηρεσίες Microsoft Online Services και στο cloud.

## <a name="define-an-integration-strategy"></a>Ορίστε τη στρατηγική μια ενοποίηση
Η Microsoft έχει τρία σενάρια κύριο ενοποίησης που είναι ταυτότητες cloud, συγχρονισμένη ταυτότητες και ομόσπονδη ταυτότητες.  Πρέπει να σχεδιάσετε σε μία από αυτές τις στρατηγικές ενοποίησης έγκριση.  Η στρατηγική που επιλέγετε μπορεί να διαφέρει και τις αποφάσεις σε μία επιλογή μπορεί να περιλαμβάνει, τον τύπο της εμπειρίας χρήστη που θέλετε να παρέχετε, έχετε ορισμένες από την υπάρχουσα υποδομή ήδη στην ίδια θέση και ποια είναι η πιο οικονομική.  
 
![](./media/hybrid-id-design-considerations/integration-scenarios.png)

Τα σενάρια που ορίζονται από το στην παραπάνω εικόνα είναι:

- **Ταυτότητες cloud**: αυτές είναι ταυτότητες που υπάρχουν αποκλειστικά στο cloud.  Στην περίπτωση Azure AD, θα βρίσκονται συγκεκριμένα στον κατάλογό σας Azure AD.
- **Συγχρονισμένες**: αυτές είναι ταυτότητες που υπάρχουν εσωτερικής εγκατάστασης και στο cloud.  Χρήση Azure AD Connect, αυτοί οι χρήστες είτε δημιουργούνται ή συνδεθεί με υπαρχόντων λογαριασμών Azure AD.  Κατακερματισμός τον κωδικό πρόσβασης του χρήστη συγχρονίζεται από το περιβάλλον εσωτερικής εγκατάστασης στο cloud στο αυτό που ονομάζεται κατακερματισμός τον κωδικό πρόσβασης.  Όταν χρησιμοποιείτε συγχρονιστεί το ότι είναι ότι εάν ένας χρήστης είναι απενεργοποιημένη στο περιβάλλον εσωτερικής εγκατάστασης, αυτό μπορεί να διαρκέσει έως 3 ώρες για αυτήν την κατάσταση λογαριασμού ώστε να εμφανίζεται στο Azure AD.  Αυτό οφείλεται το χρονικό διάστημα συγχρονισμού.
- **Ομόσπονδων**: αυτών των ταυτοτήτων υπάρχουν δύο εσωτερικής εγκατάστασης και στο cloud.  Χρήση Azure AD Connect, αυτοί οι χρήστες είτε δημιουργούνται ή συνδεθεί με υπαρχόντων λογαριασμών Azure AD.

## <a name="define-synchronization-strategy"></a>Ορίστε τη στρατηγική συγχρονισμού
Σε αυτήν την εργασία θα μπορείτε να καθορίσετε τα εργαλεία που θα χρησιμοποιηθεί για το συγχρονισμό της εταιρείας εσωτερικής δεδομένων στο cloud και τι πρέπει να χρησιμοποιήσετε τοπολογίας.  Επειδή οι περισσότερες εταιρείες χρησιμοποιήσετε υπηρεσίας καταλόγου Active Directory, πληροφορίες σχετικά με τη χρήση Azure AD Connect διεύθυνση τις παραπάνω ερωτήσεις παρέχεται με ορισμένες λεπτομέρειες.  Για περιβάλλοντα που δεν έχουν υπηρεσίας καταλόγου Active Directory, υπάρχει κάποια πληροφορία σχετικά με τη χρήση FIM 2010 R2 ή MIM 2016 για να σας βοηθήσει Σχεδιασμός αυτήν τη στρατηγική.  Ωστόσο, μελλοντικές εκδόσεις του Azure AD Connect θα υποστηρίζει καταλόγους LDAP, επομένως, ανάλογα με τη λωρίδα χρόνου, αυτές οι πληροφορίες ενδέχεται να μπορείτε να σας βοηθήσουν.

###<a name="synchronization-tools"></a>Εργαλεία συγχρονισμού
Με την πάροδο του χρόνου, διάφορα εργαλεία συγχρονισμού έχουν υπήρχαν και να χρησιμοποιηθεί για διάφορα σενάρια.  Προς το παρόν Azure AD Connect είναι στο εργαλείο της επιλογής για όλα τα υποστηριζόμενα σενάρια.  AAD Sync και DirSync επίσης εξακολουθεί να είναι γύρω από και μπορεί να ακόμα να υπάρχουν στο περιβάλλον σας τώρα. 

>[AZURE.NOTE]
Για τις πιο πρόσφατες πληροφορίες σχετικά με τις υποστηριζόμενες δυνατότητες κάθε εργαλείου, διαβάστε το άρθρο [σύγκριση εργαλεία ενοποίησης καταλόγου](active-directory-hybrid-identity-design-considerations-tools-comparison.md) .  

### <a name="supported-topologies"></a>Υποστηριζόμενες τοπολογίες
Όταν ορίζετε μια στρατηγική συγχρονισμού, πρέπει να καθοριστούν της τοπολογίας που χρησιμοποιείται. Ανάλογα με τις πληροφορίες που καθορίστηκε στο βήμα 2, μπορείτε να προσδιορίσετε ποια τοπολογία είναι το πρώτο γράμμα κάθε λέξης για να χρησιμοποιήσετε. Μεμονωμένο σύμπλεγμα, μία τοπολογία Azure AD είναι οι πιο συνηθισμένες και αποτελείται από ένα μεμονωμένο δάσος υπηρεσίας καταλόγου Active Directory και μια μεμονωμένη περίοδο λειτουργίας του Azure AD.  Αυτό πρόκειται να χρησιμοποιηθεί σε ένα μεγαλύτερο μέρος των τα σενάρια και είναι το αναμενόμενο τοπολογία κατά τη χρήση Azure AD σύνδεση Express εγκατάστασης, όπως φαίνεται στην παρακάτω εικόνα.
 
![](./media/hybrid-id-design-considerations/single-forest.png)Σενάριο μονό δάσος είναι πολύ συνηθισμένο για μεγάλες και ακόμη και σε μικρές επιχειρήσεις να έχετε πολλαπλές δομές, όπως φαίνεται στην εικόνα 5.

>[AZURE.NOTE]
Για περισσότερες πληροφορίες σχετικά με τις διαφορετικές εσωτερικής εγκατάστασης και τοπολογίες Azure AD με Azure AD Connect συγχρονισμού, διαβάστε το άρθρο [τοπολογίες για Azure AD Connect](active-directory-aadconnect-topologies.md).


![](./media/hybrid-id-design-considerations/multi-forest.png) 

Σενάριο πολλών δάσος

Εάν αυτή την περίπτωση, στη συνέχεια, την τοπολογία πολλών-forest-μονά Azure AD πρέπει να θεωρούνται εάν ισχύουν τα ακόλουθα στοιχεία:

- Οι χρήστες έχουν μόνο 1 ταυτότητας μεταξύ συμπλεγμάτων δομών όλα – την παρακάτω ενότητα με μοναδικό τρόπο αναγνώρισης χρήστες αυτό περιγράφει με περισσότερες λεπτομέρειες.
- Ο χρήστης που πραγματοποιεί έλεγχο ταυτότητας για το σύμπλεγμα στο οποίο βρίσκεται την ταυτότητα
- UPN και αγκύρωσης προέλευσης (αμετάβλητες id) θα προέρχονται από αυτόν τον τομέα
- Συμπλεγμάτων δομών όλα είναι προσβάσιμα από Azure AD Connect – αυτό σημαίνει ότι δεν χρειάζεται να είναι τομέα συνδεθεί και μπορούν να τοποθετηθούν σε μια DMZ, εάν αυτό διευκολύνει αυτό.
- Οι χρήστες έχουν μόνο ένα γραμματοκιβώτιο
- Σύμπλεγμα δομών που φιλοξενεί το γραμματοκιβώτιο χρήστη έχει την καλύτερη δυνατή ποιότητα δεδομένα για τα χαρακτηριστικά που είναι ορατές στο το Exchange καθολική λίστα διευθύνσεων (GAL)
- Εάν δεν υπάρχει καμία γραμματοκιβώτιο στο χρήστη, στη συνέχεια, οποιαδήποτε δάσος μπορεί να χρησιμοποιηθεί για να συνεισφέρετε αυτές τις τιμές
- Εάν έχετε ένα συνδεδεμένο γραμματοκιβώτιο, στη συνέχεια, υπάρχει επίσης έναν άλλο λογαριασμό σε διαφορετικό δάσος χρησιμοποιείται για να πραγματοποιήσετε είσοδο.

>[AZURE.NOTE]
Αντικείμενα που υπάρχουν σε δύο εσωτερικής εγκατάστασης και στο cloud "συνδέονται" μέσω ένα μοναδικό αναγνωριστικό. Στο πλαίσιο του συγχρονισμού καταλόγου, αυτό το μοναδικό αναγνωριστικό αναφέρεται ως το SourceAnchor. Στο πλαίσιο της καθολικής σύνδεσης, αυτό αναφέρεται ως το ImmutableId. [Βασικές αρχές σχεδίασης για Azure AD Connect](active-directory-aadconnect-design-concepts.md#sourceanchor) για περισσότερα θέματα σχετικά με τη χρήση των SourceAnchor.

Εάν τα παραπάνω δεν είναι αληθής και έχετε περισσότερα από ένα ενεργό λογαριασμό ή περισσότερα από ένα γραμματοκιβώτιο, Azure AD Connect θα επιλέξτε ένα και να παραβλέψετε την άλλη.  Εάν έχετε συνδέσει γραμματοκιβώτια αλλά άλλος λογαριασμός, δεν θα γίνει εξαγωγή αυτούς τους λογαριασμούς για Azure AD και αυτός ο χρήστης δεν θα είναι ένα μέλος από τις ομάδες.  Αυτό είναι διαφορετικό από τον τρόπο ήταν στο παρελθόν με DirSync και είναι σκόπιμες για καλύτερη υποστήριξη αυτά τα σενάρια πολλών δάσος. Ένα σενάριο πολλών δάσος εμφανίζεται στην παρακάτω εικόνα.
 
![](./media/hybrid-id-design-considerations/multiforest-multipleAzureAD.png) 
 
**Πολλαπλή δάσος σενάριο πολλών Azure AD**

Συνιστάται να υπάρχει μόνο ένα μεμονωμένο κατάλογο στο Azure AD για μια εταιρεία, αλλά υποστηρίζεται μια σχέση 1:1 διατηρείται μεταξύ ενός διακομιστή συγχρονισμού Azure AD Connect και ενός καταλόγου Azure AD.  Για κάθε παρουσία του Azure AD, θα χρειαστείτε μια εγκατάσταση του Azure AD Connect.  Επίσης, Azure AD, με σχεδίαση είναι απομόνωσης και οι χρήστες σε μία παρουσία του Azure AD δεν θα μπορούν να βλέπουν οι χρήστες σε μια άλλη παρουσία.

Είναι δυνατή και υποστηρίζεται για να συνδεθείτε μία παρουσία εσωτερικής εγκατάστασης της υπηρεσίας καταλόγου Active Directory με πολλούς καταλόγους Azure AD, όπως φαίνεται στην παρακάτω εικόνα:

![](./media/hybrid-id-design-considerations/single-forest-flitering.png) 
 

**Σενάριο φιλτραρίσματος μίας δάσος**

Για να κάνετε τα εξής πρέπει να είναι αληθή:

- Azure AD Connect συγχρονισμού διακομιστές πρέπει να ρυθμιστεί για το φιλτράρισμα, ώστε να έχουν ένα σύνολο αμοιβαία αποκλειόμενων αντικειμένων.  Αυτήν την εργασία σας, για παράδειγμα, με την εμβέλεια κάθε διακομιστή σε ένα συγκεκριμένο τομέα ή την οργανική Μονάδα.
- Ένας τομέας DNS μπορεί να καταχωρηθεί μόνο σε μία καταλόγου Azure AD έτσι ώστε το UPN των χρηστών σε την εσωτερική AD πρέπει να χρησιμοποιήσετε ξεχωριστά πεδία ονομάτων
- Οι χρήστες σε μία παρουσία του Azure AD μόνο θα μπορούν να βλέπουν οι χρήστες από την παρουσία.  Δεν θα μπορούν να βλέπουν οι χρήστες άλλων περιόδων
- Μόνο για έναν από τους καταλόγους Azure AD να ενεργοποιήσετε την υβριδική ανάπτυξη του Exchange με το εσωτερικής AD
- Αμοιβαία αποκλειστικότητα ισχύει επίσης για write-back.  Έτσι, ορισμένες δυνατότητες write-back δεν υποστηρίζονται με αυτό τοπολογία εφόσον αυτά προϋποθέτουν μια ρύθμιση παραμέτρων μόνο εσωτερικής εγκατάστασης.  Αυτό περιλαμβάνει:
 - Ομαδοποίηση write-back με προεπιλεγμένες ρυθμίσεις
 - Συσκευή write-back


Λάβετε υπόψη ότι τα ακόλουθα δεν υποστηρίζονται και δεν θα πρέπει να επιλέγονται ως μια εφαρμογή:

- Δεν υποστηρίζεται για να έχετε πολλούς διακομιστές συγχρονισμού Azure AD Connect τη σύνδεση με το ίδιο καταλόγου Azure AD, ακόμα και αν να έχουν ρυθμιστεί για να συγχρονίσετε σύνολο αμοιβαία αποκλειόμενων του αντικειμένου
- Αυτό δεν υποστηρίζεται για να συγχρονίσετε αυτόν το χρήστη να πολλών Azure AD καταλόγων. 
- Αυτό είναι επίσης που δεν υποστηρίζονται για να κάνετε μια αλλαγή για να κάνετε τους χρήστες σε μία Azure AD να εμφανίζονται ως επαφές στο άλλο καταλόγου Azure AD ρύθμισης παραμέτρων. 
- Είναι επίσης δεν υποστηρίζεται για να τροποποιήσετε Azure AD Connect συγχρονισμού για να συνδεθείτε με πολλούς καταλόγους Azure AD.
- Azure AD κατάλογοι είναι με απομονωμένες σχεδίασης. Αυτό δεν υποστηρίζεται για να αλλάξετε τη ρύθμιση παραμέτρων του Azure AD Connect συγχρονισμός με την ανάγνωση δεδομένων από μια άλλη καταλόγου Azure AD σε μια προσπάθεια για να δημιουργήσετε μια κοινές και ενοποιημένου GAL μεταξύ των καταλόγων. Επίσης δεν υποστηρίζεται για να εξαγάγετε χρήστες ως επαφών σε ένα άλλο εσωτερικής AD χρησιμοποιώντας Azure AD Connect συγχρονισμού.


>[AZURE.NOTE]
Εάν η εταιρεία σας περιορίζει υπολογιστές στο δίκτυό σας από τη σύνδεση στο Internet, αυτό το άρθρο παραθέτει τα τελικά σημεία (FQDN IPv4 και IPv6 περιοχές διευθύνσεων) που θα πρέπει να συμπεριλάβετε στο σας εξερχομένων επιτρέπεται σε λίστες και Internet Explorer ζώνη αξιόπιστων τοποθεσιών του προγράμματος-πελάτη υπολογιστές για να βεβαιωθείτε ότι υπολογιστές σας με επιτυχία χρήσης του Office 365. Για περισσότερες πληροφορίες, διαβάστε [διευθύνσεις URL του Office 365 και περιοχές διευθύνσεων IP](https://support.office.com/article/Office-365-URLs-and-IP-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2?ui=en-US&rs=en-US&ad=US).

## <a name="define-multi-factor-authentication-strategy"></a>Ορίστε τη στρατηγική ελέγχου ταυτότητας πολλαπλών παραγόντων
Σε αυτήν την εργασία που θα καθορίσει τη στρατηγική ελέγχου ταυτότητας πολλαπλών παραγόντων για να χρησιμοποιήσετε.  Azure έλεγχο ταυτότητας πολλών παραγόντων είναι δύο διαφορετική έκδοση.  Ένα είναι μια βασισμένη στο cloud και το άλλο είναι εσωτερικής εγκατάστασης με βάση χρησιμοποιώντας το Azure διακομιστή MFA.  Με βάση την αξιολόγηση μπορείτε παραπάνω μπορείτε να προσδιορίσετε ποια είναι η λύση τον σωστό τομέα για της στρατηγικής.  Χρησιμοποιήστε τον παρακάτω πίνακα για να προσδιορίσετε ποια επιλογή σχεδίαση καλύτερη εκπληρώσει απαίτηση ασφαλείας της εταιρείας σας:

Σχεδίαση πολλών παραγόντων επιλογές:

| Περιουσιακού στοιχείου για την ασφάλιση                                               | MFA στο cloud | MFA εσωτερική εγκατάσταση |
|---------------------------------------------------------------|------------------|----------------|
| Εφαρμογές της Microsoft                                                | Ναι              | Ναι            |
| Εφαρμογές ΑΔΑ στη συλλογή εφαρμογών                                  | Ναι              | Ναι            |
| Εφαρμογές των υπηρεσιών IIS που δημοσιεύονται μέσω διακομιστή μεσολάβησης εφαρμογής AD Azure         | Ναι              | Ναι            |
| Εφαρμογές των υπηρεσιών IIS που δεν δημοσιεύονται μέσω του διακομιστή μεσολάβησης Azure AD εφαρμογής | Όχι               | Ναι            |
| Απομακρυσμένη πρόσβαση ως VPN, RDG                                     | Όχι               | Ναι            |

Παρόλο που μπορεί να έχουν τακτοποιηθεί σε μια λύση για τη στρατηγική, πρέπει ακόμα να χρησιμοποιήσετε την αξιολόγηση από επάνω σε όπου βρίσκονται οι χρήστες σας.  Αυτό μπορεί να προκαλέσει τη λύση για να αλλάξετε.  Χρησιμοποιήστε τον παρακάτω πίνακα για να σας βοηθήσουν να Καθορισμός αυτό:

| Θέση χρήστη                                                       | Επιλογή προτιμώμενη σχεδίασης                 |
|---------------------------------------------------------------------|-----------------------------------------|
| Καταλόγου Azure Active Directory                                              | Πολλούς-FactorAuthentication στο cloud |
| Azure AD και εσωτερικής εγκατάστασης με χρήση Ομοσπονδία με AD FS AD             | Και τα δύο                                    |
| Azure AD και εσωτερικής AD χρησιμοποιώντας Azure AD Connect χωρίς συγχρονισμό κωδικού πρόσβασης | Και τα δύο                                    |
| Azure AD και εσωτερικής εγκατάστασης με χρήση Azure AD Connect με το συγχρονισμό κωδικού πρόσβασης  | Και τα δύο                                    |
| Εσωτερικής εγκατάστασης AD                                                      | Διακομιστής ελέγχου ταυτότητας πολλαπλών παραγόντων      |

>[AZURE.NOTE]
Πρέπει επίσης να διασφαλίζετε ότι η επιλογή σχεδίαση έλεγχο ταυτότητας πολλών παραγόντων που έχετε επιλέξει υποστηρίζει τις δυνατότητες που απαιτούνται για τη σχεδίαση.  Για περισσότερες πληροφορίες, διαβάστε [επιλογή της λύσης πολλαπλών παραγόντων ασφαλείας για εσάς](../multi-factor-authentication-get-started.md#what-am-i-trying-to-secure).

## <a name="multi-factor-auth-provider"></a>Υπηρεσία παροχής ελέγχου ταυτότητας πολλαπλών παραγόντων
Έλεγχος ταυτότητας πολλών παραγόντων είναι διαθέσιμη από προεπιλογή για καθολικοί διαχειριστές που έχουν ένα μισθωτή Azure Active Directory. Ωστόσο, εάν θέλετε να επεκτείνετε τον έλεγχο ταυτότητας πολλών παραγόντων για όλους τους χρήστες σας ή/και θέλετε να σας καθολικοί διαχειριστές να είναι σε θέση να εκμεταλλευτείτε τις δυνατότητες όπως η πύλη διαχείρισης, προσαρμοσμένες χαιρετισμών και τις αναφορές, στη συνέχεια, πρέπει να αγοράσετε και ρύθμιση παραμέτρων της υπηρεσίας παροχής ελέγχου ταυτότητας πολλαπλών παραγόντων.

>[AZURE.NOTE]
Πρέπει επίσης να διασφαλίζετε ότι η επιλογή σχεδίαση έλεγχο ταυτότητας πολλών παραγόντων που έχετε επιλέξει υποστηρίζει τις δυνατότητες που απαιτούνται για τη σχεδίαση.
