#<a name="implement-mobile-engagement-with-media-app"></a>Υλοποίηση της δέσμευσης κινητές συσκευές με εφαρμογή πολυμέσων

## <a name="overview"></a>Επισκόπηση

Γιάννης είναι διαχειριστής έργου κινητές συσκευές για μια εταιρεία που μεγάλο πολυμέσων. Εσείς πρόσφατα ξεκινά μια νέα εφαρμογή που έχει μια καταμέτρηση πολύ υψηλή λήψης. Αυτός περιλαμβάνει πατήσετε την επίτευξη των στόχων για τη λήψη, αλλά εξακολουθούν να του επιστρέψετε στην Investment(ROI) ανά χρήστη δεν πληροί ανάγκες του. 

Ιωάννης έχει ήδη που έχουν προσδιοριστεί για ποιο λόγο το απόδοση Επένδυσης είναι πολύ χαμηλή. Οι χρήστες συχνά διακόψετε τη χρήση του app μετά μόνο 2 εβδομάδες και τους ποτέ προκύπτουν ξανά. Θέλει να αυξήσετε τη διατήρηση της εφαρμογής του.

Αφού ορισμένα αρχική δοκιμές, αυτός περιλαμβάνει μάθατε όταν άσκηση του οι χρήστες με τις ειδοποιήσεις push, τους την τάση να συνεχίσετε να χρησιμοποιείτε την εφαρμογή. Επίσης τους χρήστες που έχουν ανενεργός συχνά θα επιστρέψει στην εφαρμογή ανάλογα με τις ειδοποιήσεις εσείς τους αποστέλλεται. John αποφασίσει να επενδύσει σε κάποιο είδος δέσμευση πρόγραμμα για την εφαρμογή που χρησιμοποιεί για προχωρημένους στόχευσης με τις ειδοποιήσεις push.

Ιωάννης έχει πρόσφατα Διαβάστε τη [Δέσμευση Mobile Azure - οδηγού για γρήγορα αποτελέσματα με τις βέλτιστες πρακτικές](mobile-engagement-getting-started-best-practices.md) και έχει αποφασίσει να υλοποιήσετε τις συστάσεις από τον οδηγό.

##<a name="objectives-and-kpis"></a>Στόχοι και KPI

Βασικοί ενδιαφερόμενοι για την εφαρμογή του John πληροί. Επιχειρήσεις δημιουργείται από διαφημίσεις, όπως χρήστες εκμετάλλευση του πολυμέσων. Αυξάνοντας το περιεχόμενο που καταναλώθηκε ανά χρήστη, John αυξάνει τον έσοδα. Συμφωνήσετε πάνω σε ένα κύριο στόχο: για να αυξήσετε τις πωλήσεις από διαφημίσεις κατά 25%. Δημιουργούν επιχειρήσεις βασικών δεικτών απόδοσης (KPI) για να μέτρησης και της μονάδας δίσκου αυτός ο στόχος

* Αριθμός διαφημίσεων κλικ ανά χρήστη
* Πόσες σελίδων άρθρων που επισκεφτήκατε (ανά χρήστη / ανά περίοδο λειτουργίας / ανά εβδομάδα / ανά μήνα...)
* Τι είναι οι αγαπημένες κατηγορίες

Που βασίζονται σε σύσκεψη του John με βασικοί ενδιαφερόμενοι αυτός έχει οριστεί του KPI επιχειρήσεις. Αυτός ακολουθεί μέρος 1 από το [Azure Mobile δέσμευση - οδηγού για γρήγορα αποτελέσματα με τις βέλτιστες πρακτικές](mobile-engagement-getting-started-best-practices.md). 

Στη συνέχεια, δημιουργεί τα παρακάτω KPI δέσμευση για να βεβαιωθείτε ότι έχουν συμπληρωθεί στόχοι:

* Παρακολούθηση της διατήρησης σε τα παρακάτω διαστήματα: ημερήσια, εβδομαδιαία, δύο φορές την εβδομάδα και μηνιαία.
* Καταμετρά ενεργοί χρήστες
* Αποθηκεύει το χαρακτηρισμό εφαρμογής στην εφαρμογή

Το παρακάτω τεχνική KPI που βασίζονται σε συστάσεις από την ομάδα, έχουν προστεθεί στο απαντούν στα παρακάτω ερωτήματα:

* Τι είναι η διαδρομή μου χρήστη (επισκέπτεται ποια σελίδα, τον αριθμό των χρηστών ώρα αφιερώνετε σε αυτό)
* Αριθμός παρουσιάζει σφάλμα ή σφάλματα αντιμετώπισε ανά περίοδο λειτουργίας;
* Ποιες εκδόσεις OS εκτελούν οι χρήστες μου;
* Τι είναι το μέσο μέγεθος της οθόνης για τους χρήστες μου;
* Τι είδους συνδέσεις στο internet έχουν τους χρήστες μου;

Για κάθε δείκτη KPI, εσείς ταξινομεί τα δεδομένα που απαιτούνται και εσείς εγγραφών στην κατάλληλη θέση των δικών playbook.

## <a name="engagement-program-and-integration"></a>Δέσμευση προγράμματος και ενοποίηση

Τώρα που John ολοκληρωθεί τον ορισμό του KPI, αρχίζει του φάση στρατηγικής δέσμευσης, καθορίζοντας 4 δέσμευση προγράμματα και τους στόχους τους:    ![][1]

Στη συνέχεια, John μεταβαίνει βαθύτερη, που περιγράφει με λεπτομέρεια τις ειδοποιήσεις push για κάθε πρόγραμμα. Ειδοποιήσεων Push καθορίζονται από πέντε στοιχεία:

1. Στόχος: Τι είναι ο στόχος της ειδοποίησης
2. Πώς θα είναι δυνατή η πρόσβαση του στόχου
3. Στόχος: που θα λάβουν την ειδοποίηση;
4. Content: Τι είναι το κείμενο και τη μορφή της ειδοποίησης (στο App/Out της εφαρμογής)
5. Πότε: Τι είναι το καλύτερη στιγμή για να στείλετε αυτήν την ειδοποίηση push

    ![][2]

Για περισσότερες πληροφορίες ανατρέξτε το [Playbooks](https://github.com/Azure/azure-mobile-engagement-samples/tree/master/Playbooks).

Σύμφωνα με το τμήμα 2 από τη λευκή βίβλο John χρησιμοποιεί ενότητα προορισμού για να προσδιορίσετε ποια δεδομένα πρέπει να συλλέξετε και εγγράφει την ετικέτα Σχεδιασμός από κοινού με την ομάδα IT για την υλοποίηση της λύσης. Μετά από 1 εβδομάδα της εφαρμογής και δοκιμή αποδοχής χρήστη, John τέλος να εκκινήσετε τα προγράμματα.

##<a name="program-results"></a>Αποτελέσματα του προγράμματος

4 μήνες αργότερα, John εξετάζει επιδόσεις των προγραμμάτων. Το πρόγραμμα Καλώς ορίσατε και το Εβδομαδιαίο πρόγραμμα σύσκεψη για την επίτευξη των στόχων. Μείωση του αριθμού των χρηστών με μία μόνο περίοδο λειτουργίας, που χρησιμοποιούνται περισσότερες δυνατότητες της εφαρμογής και ο αριθμός των συνδέσεων ανά εβδομάδα έχει διπλό.

Το **Πρόγραμμα ανενεργός** βοηθάει John Κατανόηση τάσεις χρήστη. Φαίνεται ότι το 15% των χρηστών που είναι ανενεργοί επιστρέψτε στην εφαρμογή. Ωστόσο οι περισσότερες από αυτές δεν παραμένουν ενεργό περισσότερες από 1 μήνα. John προβλέπει μια πιθανή βελτιστοποίηση από αυτήν τη σειρά με τις πρόσθετες ειδοποιήσεις και επέκταση του περιεχομένου επιλογές.

**Ανακαλύψτε το πρόγραμμα** δεν λειτουργεί σωστά. Αυξάνει διασταύρωσης πώλησης αλλά δεν επαρκεί για την επίτευξη των στόχων του. John προσδιορίζει ότι εσείς δεν διαθέτει αρκετά δεδομένα για να κάνετε σχετικές στόχευσης και προτείνει κατάλληλο περιεχόμενο. Ο ίδιος σταματά αυτού του προγράμματος και εστιάζει σχετικά με την αποστολή "σύνταξης τις ειδοποιήσεις push" με δέσμευση Mobile Azure. Προσωπική δημοσιογράφοι έχει ήδη μια λύση CMS για την αποστολή ειδοποιήσεων push και αυτές δεν θέλετε να αλλάξετε.

John αποφασίσει να χρησιμοποιήσετε το API επίτευξη το οποίο είναι ένα HTTP REST API που επιτρέπει τη Διαχείριση εκστρατείες Reach χωρίς να χρειάζεται να χρησιμοποιήσετε περιβάλλον AZME Web. Με αυτήν την προσέγγιση John μπορεί να συλλέξει τα δεδομένα ο ίδιος ανάγκες και να επιτρέπεται ο στους συντάκτες να συνεχίσετε να χρησιμοποιείτε τη λύση CMS.

Για να βεβαιωθείτε ότι αυτή η δυνατότητα λειτουργεί σωστά, John ζητά από την ομάδα IT για να είστε προσεκτικοί για τα ακόλουθα θέματα:

1. **Η λειτουργία συστήματα** : όλα έχουν τις δικές τους κανόνες για να διαχειριστείτε τις ειδοποιήσεις push, ώστε να John καθορίζεται σε λίστα όλες τις περιπτώσεις και ελέγχει εάν τα API χειρίζεται αυτό.
Παράδειγμα: Σύστημα Android push επιτρέπει γενική εικόνα που δεν συμβαίνει με iOS.

2. **Χρονικό πλαίσιο**: John θέλει ένα API, το οποίο ορίσουν το χρονικό πλαίσιο και ορίστε τέρμα σε εκστρατείες. Θέλει να διατηρηθούν χρήστες από οποιαδήποτε bombing ενοχλητικούς ειδοποίησης.

3. **Κατηγορίες**: ομάδας μάρκετινγκ προετοιμάζει το πρότυπο για κάθε τύπο ειδοποίησης. John ζητά από την ομάδα IT για να ορίσετε κατηγορίες εντός του API.

Μετά από ορισμένες δοκιμές John είναι ικανοποιημένοι. Ευχαριστήριο αυτό το API, δημοσιογράφοι μπορεί να στείλει τις ειδοποιήσεις push με τους CMS και δέσμευση Mobile Azure συλλέγει όλα τα δεδομένα συμπεριφοράς τους

Μετά από αυτές τις 4 πρώτα μήνες, αποτελέσματα αντικατοπτρίζουν μια καλή συνολική απόδοση και παρέχει εμπιστοσύνης για John και του πίνακα, για την απόδοση της Επένδυσης ανά χρήστη αυξάνεται ανά 15% και κινητές συσκευές πωλήσεων αναπαριστούν 17,5% των συνολικών πωλήσεων, αύξηση 7.5% μόνο τέσσερις μήνες.
