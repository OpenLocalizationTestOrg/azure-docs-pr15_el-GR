# <a name="azure-search-service-rest-api-version-2015-02-28-preview"></a>Αναζήτηση Azure Service REST API: 2015 έκδοση-02-28-Preview

Σε αυτό το άρθρο είναι η τεκμηρίωση αναφοράς για `api-version=2015-02-28-Preview`. Αυτή η προεπισκόπηση επεκτείνει την τρέχουσα έκδοση γενικά διαθέσιμο, [έκδοση api = 2015-02-28](https://msdn.microsoft.com/library/dn798935.aspx), παρέχοντας οι παρακάτω δυνατότητες πειραματικής:

- `moreLikeThis`ερώτημα παραμέτρων σε την [Αναζήτηση εγγράφων](#SearchDocs) API. Εντοπίζει άλλα έγγραφα που έχουν σχέση με ένα άλλο συγκεκριμένο έγγραφο.

Μερικά επιπλέον τμήματα του `2015-02-28-Preview` REST API τεκμηριώνονται ξεχωριστά. Περιλαμβάνουν τα εξής:

- [Βαθμολογίας προφίλ](search-api-scoring-profiles-2015-02-28-preview.md)
- [Οι δεικτοδότες](search-api-indexers-2015-02-28-preview.md)

Azure υπηρεσία αναζήτησης είναι διαθέσιμες σε πολλές εκδόσεις. Ανατρέξτε σε [Διαχείριση εκδόσεων υπηρεσίας αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) για λεπτομέρειες.

<a name="IndexOps"></a>
## <a name="index-operations"></a>Λειτουργίες ευρετηρίου

Μπορείτε να δημιουργήσετε και να διαχειριστείτε τα ευρετήρια σε υπηρεσία αναζήτησης Azure μέσω απλό αιτήσεις HTTP (ΔΗΜΟΣΊΕΥΣΗ, ΛΉΨΗ, ΤΟΠΟΘΈΤΗΣΗ, ΔΙΑΓΡΑΦΉ) σε σχέση με έναν πόρο που δίνεται ευρετηρίου. Για να δημιουργήσετε ένα ευρετήριο, μπορείτε πρώτα να ΔΗΜΟΣΙΕΎΣΕΤΕ ένα έγγραφο JSON που περιγράφει το σχήμα ευρετηρίου. Το σχήμα ορίζει τα πεδία από το ευρετήριο, τους τύπους δεδομένων και πώς μπορεί να χρησιμοποιηθεί (για παράδειγμα, σε αναζητήσεις πλήρους κειμένου, φίλτρα, ταξινόμηση ή faceting). Καθορίζει επίσης βαθμολογίας προφίλ, suggesters και άλλα χαρακτηριστικά για να ρυθμίσετε τη συμπεριφορά του ευρετηρίου.

Το παρακάτω παράδειγμα παρέχει μια εικόνα από ένα σχήμα που χρησιμοποιείται για την αναζήτηση σε ξενοδοχείο πληροφορίες με το πεδίο Περιγραφή που ορίζεται σε δύο γλώσσες. Παρατηρήστε πώς χαρακτηριστικά ελέγξετε τον τρόπο χρήσης του πεδίου. Για παράδειγμα το `hotelId` χρησιμοποιείται ως το κλειδί εγγράφου (`"key": true`) και εξαιρείται από τις αναζητήσεις πλήρους κειμένου (`"searchable": false`).

    {
    "name": "hotels",  
    "fields": [
      {"name": "hotelId", "type": "Edm.String", "key": true, "searchable": false},
      {"name": "baseRate", "type": "Edm.Double"},
      {"name": "description", "type": "Edm.String", "filterable": false, "sortable": false, "facetable": false},
      {"name": "description_fr", "type": "Edm.String", "filterable": false, "sortable": false, "facetable": false, "analyzer": "fr.lucene"},
      {"name": "hotelName", "type": "Edm.String"},
      {"name": "category", "type": "Edm.String"},
      {"name": "tags", "type": "Collection(Edm.String)"},
      {"name": "parkingIncluded", "type": "Edm.Boolean"},
      {"name": "smokingAllowed", "type": "Edm.Boolean"},
      {"name": "lastRenovationDate", "type": "Edm.DateTimeOffset"},
      {"name": "rating", "type": "Edm.Int32"},
      {"name": "location", "type": "Edm.GeographyPoint"}
     ],
     "suggesters": [
      {
       "name": "sg",
       "searchMode": "analyzingInfixMatching",
       "sourceFields": ["hotelName"]
      }
     ]
    }

Αφού δημιουργήσετε το ευρετήριο, θα αποστείλετε έγγραφα που συμπληρώνουν το ευρετήριο. Ανατρέξτε στο θέμα [Προσθήκη ή ενημέρωση εγγράφων](#AddOrUpdateDocuments) για αυτό το επόμενο βήμα.

Για ένα βίντεο εισαγωγής για τη δημιουργία ευρετηρίου στο πλαίσιο Αναζήτηση Azure, ανατρέξτε στο θέμα το [κανάλι 9 Cloud εξώφυλλο επεισοδίου στην αναζήτηση Azure](http://go.microsoft.com/fwlink/p/?LinkId=511509).


<a name="CreateIndex"></a>
## <a name="create-index"></a>Δημιουργία ευρετηρίου

Ένα ευρετήριο είναι το κύριο μέσο οργάνωση και αναζήτηση εγγράφων στο Azure αναζήτηση, παρόμοια με τον τρόπο πίνακα οργανώνουν εγγραφές σε μια βάση δεδομένων. Κάθε ευρετήριο έχει μια συλλογή εγγράφων που όλα συμμορφώνεται με το σχήμα ευρετηρίου (ονόματα πεδίων, τους τύπους δεδομένων και ιδιότητες), αλλά τα ευρετήρια επίσης καθορίσετε επιπλέον τα δομές (suggesters βαθμολογίας προφίλ και επιλογές CORS) που ορίζουν άλλες συμπεριφορές αναζήτησης.

Μπορείτε να δημιουργήσετε ένα νέο ευρετήριο μέσα σε μια υπηρεσία αναζήτησης Azure χρησιμοποιώντας μια αίτηση HTTP POST ή ΤΟΠΟΘΈΤΗΣΗ. Στο σώμα της πρόσκλησης σε είναι ένα σχήμα JSON που καθορίζει τις πληροφορίες ευρετηρίου και ρύθμισης παραμέτρων.

    POST https://[service name].search.windows.net/indexes?api-version=[api-version]
    Content-Type: application/json
    api-key: [admin key]

Εναλλακτικά, μπορείτε να χρησιμοποιήσετε ΤΟΠΟΘΈΤΗΣΗ και να καθορίσετε το όνομα του ευρετηρίου στο URI. Εάν δεν υπάρχει το ευρετήριο, θα δημιουργηθεί.

    PUT https://[search service url]/indexes/[index name]?api-version=[api-version]

Δημιουργία ευρετηρίου καθορίζει τη δομή των εγγράφων που έχουν αποθηκευτεί και χρησιμοποιούνται στις λειτουργίες αναζήτησης. Συμπληρώνετε το ευρετήριο είναι μια ξεχωριστή λειτουργία. Για αυτό το βήμα, μπορείτε να χρησιμοποιήσετε ένα [ευρετήριο](https://msdn.microsoft.com/library/azure/mt183328.aspx) (διαθέσιμο για υποστηριζόμενες προελεύσεις δεδομένων) ή μια λειτουργία [Προσθήκη, ενημέρωση, ή να διαγράψετε έγγραφα](https://msdn.microsoft.com/library/azure/dn798930.aspx) . Το ευρετήριο ανεστραμμένη δημιουργείται όταν τα έγγραφα δημοσιεύονται.

**Σημείωση**: Ο μέγιστος αριθμός των επιτρεπόμενων ευρετήρια διαφέρει ανάλογα με τις τιμές σε επίπεδο. Τη δωρεάν υπηρεσία επιτρέπει έως 3 ευρετήρια. Βασική υπηρεσία επιτρέπει 50 ευρετήρια ανά υπηρεσίας αναζήτησης. Ανατρέξτε στο θέμα [όρια και περιορισμούς](http://msdn.microsoft.com/library/azure/dn798934.aspx) για λεπτομέρειες.

**Αίτηση**

HTTPS είναι απαραίτητη για όλες τις αιτήσεις εξυπηρέτησης. Η αίτηση **Create Index** είναι δυνατό να δημιουργηθεί χρησιμοποιώντας μια ΔΗΜΟΣΊΕΥΣΗ ή ΤΟΠΟΘΈΤΗΣΗ μέθοδο. Όταν χρησιμοποιείτε τη ΔΗΜΟΣΊΕΥΣΗ, πρέπει να δώσετε ένα όνομα ευρετηρίου στο σώμα της αίτησης μαζί με τον ορισμό σχήματος του ευρετηρίου. Με την ΤΟΠΟΘΈΤΗΣΗ, το όνομα του ευρετηρίου αποτελεί τμήμα της διεύθυνσης URL. Εάν δεν υπάρχει το ευρετήριο, δημιουργείται. Εάν υπάρχει ήδη, ενημερώνεται για το νέο ορισμό.

Το όνομα του ευρετηρίου πρέπει να είναι πεζά Έναρξη με ένα γράμμα ή έναν αριθμό, έχετε καμία καθέτους ή τελείες και είναι μικρότερη από 128 χαρακτήρες. Μετά την εκκίνηση το όνομα του ευρετηρίου με ένα γράμμα ή έναν αριθμό, το υπόλοιπο του ονόματος να συμπεριλάβετε οποιοδήποτε γράμμα, αριθμός και παύλες, με την προϋπόθεση ότι δεν είναι διαδοχικά το παύλες.

Το `api-version` απαιτείται. Για μια λίστα με τις διαθέσιμες εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης.

- `Content-Type`: Υποχρεωτικό. Ορίσετε την επιλογή`application/json`
- `api-key`: Υποχρεωτικό. Το `api-key` χρησιμοποιείται για
- έλεγχος ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικά στην υπηρεσία σας. Η **Δημιουργία ευρετηρίου** αίτηση πρέπει να περιλαμβάνει μια `api-key` κεφαλίδα το κλειδί διαχείρισης (σε αντίθεση με έναν αριθμό-κλειδί ερωτήματος).

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε και τις δύο το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

<a name="RequestData"></a>
**Σύνταξη σώμα αίτησης**

Στο σώμα της πρόσκλησης σε περιέχει έναν ορισμό σχήματος, που περιλαμβάνει τη λίστα των πεδίων δεδομένων μέσα σε έγγραφα που θα είναι τροφοδοτεί αυτό το ευρετήριο, τους τύπους δεδομένων, χαρακτηριστικά, καθώς και μια προαιρετική λίστα βαθμολογίας προφίλ που χρησιμοποιούνται για τη βαθμολογία τα έγγραφα που συμφωνούν στιγμή ερωτήματος.

Σημειώστε ότι για ένα αίτημα ΔΗΜΟΣΊΕΥΣΗ, πρέπει να καθορίσετε το όνομα του ευρετηρίου στο σώμα της αίτησης.

Μπορεί να υπάρξει μόνο ένα πεδίο κλειδιού στο ευρετήριο. Αυτό πρέπει να είναι ένα πεδίο συμβολοσειράς. Αυτό το πεδίο αντιπροσωπεύει το μοναδικό αναγνωριστικό για κάθε έγγραφο που είναι αποθηκευμένο σε ευρετήριο.

Τα κύρια στοιχεία του ευρετηρίου περιλαμβάνουν τα εξής:

- `name`
- `fields`που θα είναι τροφοδοτεί αυτό το ευρετήριο, συμπεριλαμβάνοντας όνομα, τύπος δεδομένων και τις ιδιότητες που Καθορισμός επιτρεπόμενων ενεργειών σε αυτό το πεδίο.
- `suggesters`χρησιμοποιείται για ερωτήματα αυτόματης καταχώρησης ή πρόβλεψης.
- `scoringProfiles`επιλογή προσαρμοσμένης αναζήτησης βαθμολογία κατάταξης. Για λεπτομέρειες, ανατρέξτε στο θέμα [Προσθήκη βαθμολογίας προφίλ](https://msdn.microsoft.com/library/azure/dn798928.aspx) .
- `analyzers`, `charFilters`, `tokenizers`, `tokenFilters` χρησιμοποιείται για να ορίσετε πώς είναι εσφαλμένος έγγραφα/τα ερωτήματά σας σε διακριτικά δυνατότητα αναζήτησης. Για λεπτομέρειες, ανατρέξτε στο θέμα [ανάλυση στο πλαίσιο Αναζήτηση Azure](https://aka.ms//azsanalysis) .
- `defaultScoringProfile`χρησιμοποιείται για να αντικαταστήσετε την προεπιλεγμένη βαθμολογίας συμπεριφορές.
- `corsOptions`Για να επιτρέψετε σε ερωτήματα σταυρό origin από το ευρετήριο.

          "searchable": true (default where applicable) | false (only Edm.String and Collection(Edm.String) fields can be searchable),

**Χαρακτηριστικά ευρετηρίου**

Τα παρακάτω χαρακτηριστικά μπορεί να οριστεί κατά τη δημιουργία ευρετηρίου. Για λεπτομέρειες σχετικά με τα προφίλ βαθμολογίας και βαθμολογίας, ανατρέξτε στο θέμα [Προσθήκη βαθμολογίας προφίλ](https://msdn.microsoft.com/library/azure/dn798928.aspx).

`name`-Ορίζει το όνομα του πεδίου.

`type`-Ορίζει τον τύπο δεδομένων για το πεδίο. Για μια λίστα των υποστηριζόμενων τύπων, ανατρέξτε στο θέμα [Τύποι δεδομένων που υποστηρίζονται](#DataTypes) .

`searchable`-Επισημαίνει το πεδίο ως πλήρους κειμένου δυνατότητα αναζήτησης. Αυτό σημαίνει ότι το θα υποβάλλονται σε ανάλυση όπως λέξεων κατά τη διάρκεια δημιουργίας ευρετηρίου. Εάν ορίσετε έναν `searchable` πεδίο σε μια τιμή όπως η "Τι καιρό day", εσωτερικά αυτό θα να διαιρεθεί σε τα μεμονωμένα διακριτικά "Τι καιρό" και "ημέρα". Αυτή η δυνατότητα επιτρέπει αναζητήσεις πλήρους κειμένου για αυτούς τους όρους. Τα πεδία του τύπου `Edm.String` ή `Collection(Edm.String)` είναι `searchable` από προεπιλογή. Δεν μπορεί να είναι τα πεδία από τους άλλους τύπους `searchable`.

  - **Σημείωση**: `searchable` πεδία εκμετάλλευση επιπλέον διάστημα του ευρετηρίου, επειδή το Azure αναζήτησης θα αποθηκεύσει μια επιπλέον κωδικοποιημένη έκδοση της τιμής του πεδίου για αναζητήσεις πλήρους κειμένου. Εάν θέλετε να εξοικονομήσετε χώρο του ευρετηρίου και δεν χρειάζεστε ένα πεδίο που θα συμπεριληφθούν στις αναζητήσεις, ορίστε `searchable` να `false`.

`filterable`-Επιτρέπει το πεδίο να αναφέρεται σε `$filter` ερωτήματα. `filterable`διαφέρει από `searchable` στο τον τρόπο χειρισμού συμβολοσειρών. Τα πεδία του τύπου `Edm.String` ή `Collection(Edm.String)` που είναι `filterable` δεν υποβάλλονται σε λέξεων, ώστε να είναι συγκρίσεις για ακριβή αντιστοιχεί μόνο. Για παράδειγμα, εάν ορίσετε ενός τέτοιου πεδίου `f` "Τι καιρό ημέρα," `$filter=f eq 'sunny'` θα βρείτε χωρίς συμφωνίες, αλλά `$filter=f eq 'sunny day'` θα. Όλα τα πεδία είναι `filterable` από προεπιλογή.

`sortable`-Από προεπιλογή το σύστημα ταξινομεί τα αποτελέσματα κατά βαθμολογία, αλλά σε πολλές εμπειρίες οι χρήστες θα θέλετε να ταξινομήσετε κατά πεδία στα έγγραφα. Τα πεδία του τύπου `Collection(Edm.String)` δεν είναι δυνατό να `sortable`. Όλα τα άλλα πεδία είναι `sortable` από προεπιλογή.

`facetable`-Συνήθως χρησιμοποιούνται σε μια παρουσίαση των αποτελεσμάτων αναζήτησης που περιλαμβάνει μέτρηση επισκέψεων κατά κατηγορία (για παράδειγμα, αναζητήστε ψηφιακές φωτογραφικές μηχανές και δείτε επισκέψεων από εμπορικής επωνυμίας, με megapixels, κατά τιμή, κ.λπ.). Αυτή η επιλογή δεν μπορούν να χρησιμοποιηθούν με πεδία τύπου `Edm.GeographyPoint`. Όλα τα άλλα πεδία είναι `facetable` από προεπιλογή.

  - **Σημείωση**: τα πεδία του τύπου `Edm.String` που είναι `filterable`, `sortable`, ή `facetable` μπορεί να είναι πολύ 32 KB μήκος. Αυτό συμβαίνει επειδή αυτά τα πεδία, υπολογίζονται ως έναν όρο αναζήτησης μόνο και το μέγιστο μήκος ενός όρου στο πλαίσιο Αναζήτηση Azure είναι 32KB. Εάν πρέπει να αποθηκεύσετε περισσότερο κείμενο από αυτό σε ένα πεδίο μία συμβολοσειρά, θα πρέπει να ορίσετε ρητά `filterable`, `sortable`, και `facetable` σε `false` στον ορισμό του ευρετηρίου.

  - **Σημείωση**: Εάν ένα πεδίο έχει κανένα από τα παραπάνω χαρακτηριστικά οριστεί `true` (`searchable`, `filterable`, `sortable`, ή`facetable`) στο πεδίο αποτελεσματική εξαιρείται από το ευρετήριο ανεστραμμένη. Αυτή η επιλογή είναι χρήσιμη για τα πεδία που δεν χρησιμοποιούνται σε ερωτήματα, αλλά είναι απαραίτητα στα αποτελέσματα αναζήτησης. Εκτός από αυτά τα πεδία από το ευρετήριο βελτιώνει τις επιδόσεις.

`key`-Επισημαίνει το πεδίο ως που περιέχει μοναδικών αναγνωριστικών για έγγραφα του ευρετηρίου. Ακριβώς ένα πεδίο πρέπει να επιλεγεί ως το `key` πεδίο και πρέπει να είναι τύπου `Edm.String`. Τα πεδία κλειδιών μπορεί να χρησιμοποιηθεί για την αναζήτηση εγγράφων απευθείας μέσω του [API αναζήτησης](#LookupAPI).

`retrievable`-Ορίζει εάν το πεδίο μπορεί να επιστραφεί στο αποτέλεσμα αναζήτησης.  Αυτό είναι χρήσιμο όταν θέλετε να χρησιμοποιήσετε ένα πεδίο (για παράδειγμα, περιθώριο) ως φίλτρο, ταξινόμησης ή βαθμολογίας μηχανισμό, αλλά δεν θέλετε το πεδίο να είναι ορατές σε του τελικού χρήστη. Αυτό το χαρακτηριστικό πρέπει να είναι `true` για `key` πεδία.

`analyzer`-Ορίζει το όνομα του την "Ανάλυση" για να χρησιμοποιήσετε για το πεδίο στο χρόνο αναζήτησης και την ώρα δημιουργίας ευρετηρίου. Για το επιτρεπόμενο σύνολο τιμών ανατρέξτε [αναλυτές](https://msdn.microsoft.com/library/mt605304.aspx). Αυτή η επιλογή μπορεί να χρησιμοποιηθεί μόνο με `searchable` πεδία και δεν μπορεί να οριστεί μαζί με ένα `searchAnalyzer` ή `indexAnalyzer`.  Αφού επιλέξετε την "Ανάλυση", δεν μπορεί να αλλάξει για το πεδίο.

`searchAnalyzer`-Ορίζει το όνομα του analyzer χρησιμοποιείται κατά το χρόνο αναζήτησης για το πεδίο. Για το επιτρεπόμενο σύνολο τιμών ανατρέξτε [αναλυτές](https://msdn.microsoft.com/library/mt605304.aspx). Αυτή η επιλογή μπορεί να χρησιμοποιηθεί μόνο με `searchable` πεδία. Πρέπει να οριστούν μαζί με `indexAnalyzer` και δεν είναι δυνατό να ρυθμιστούν μαζί με το `analyzer` την επιλογή. Αυτό ανάλυσης μπορεί να ενημερωθεί σε ένα υπάρχον πεδίο.

`indexAnalyzer`-Ορίζει το όνομα του analyzer χρησιμοποιείται κατά το χρόνο δημιουργίας ευρετηρίου για το πεδίο. Για το επιτρεπόμενο σύνολο τιμών ανατρέξτε [αναλυτές](https://msdn.microsoft.com/library/mt605304.aspx). Αυτή η επιλογή μπορεί να χρησιμοποιηθεί μόνο με `searchable` πεδία. Πρέπει να οριστούν μαζί με `searchAnalyzer` και δεν είναι δυνατό να ρυθμιστούν μαζί με το `analyzer` την επιλογή. Αφού επιλέξετε την "Ανάλυση", δεν μπορεί να αλλάξει για το πεδίο.

`suggesters`-Ορίζει τη λειτουργία αναζήτησης και τα πεδία που αποτελούν την προέλευση του περιεχομένου για προτάσεις. Για λεπτομέρειες, ανατρέξτε στο θέμα [Suggesters](#Suggesters) .

`scoringProfiles`-Καθορίζει προσαρμοσμένο βαθμολογίας συμπεριφορές που σας επιτρέπουν να επηρεάσετε ποια στοιχεία εμφανίζονται υψηλότερη στα αποτελέσματα αναζήτησης. Προφίλ βαθμολογίας αποτελούνται από τις συναρτήσεις και διαφορετικό πάχος πεδίο. Για περισσότερες πληροφορίες σχετικά με τα χαρακτηριστικά που χρησιμοποιούνται σε ένα προφίλ βαθμολογίας, ανατρέξτε στο θέμα [Προσθήκη βαθμολογίας προφίλ](https://msdn.microsoft.com/library/azure/dn798928.aspx) .

**Υποστήριξη γλωσσών**

Πεδία με δυνατότητα αναζήτησης, υποβάλλονται σε ανάλυση ώστε το μεγαλύτερο περιλαμβάνει συχνά λέξεων κανονικοποίηση κειμένου και αποκλείει τους όρους. Από προεπιλογή, τα πεδία με δυνατότητα αναζήτησης στο πλαίσιο Αναζήτηση Azure αναλύονται με την [Τυπική Lucene Apache ανάλυσης](http://lucene.apache.org/core/4_9_0/analyzers-common/index.html) που χωρίζει το κείμενο σε στοιχεία που ακολουθεί τους κανόνες["αγοράς κείμενο Unicode"](http://unicode.org/reports/tr29/) . Επιπλέον, η τυπική ανάλυση μετατρέπει όλους τους χαρακτήρες σε φόρμα τους πεζούς. Έγγραφα του ευρετηρίου και όρους αναζήτησης δούμε την ανάλυση κατά τη διάρκεια δημιουργίας ευρετηρίου και την επεξεργασία ερωτημάτων.

Azure αναζήτησης υποστηρίζει πολλές διαφορετικές γλώσσες. Κάθε γλώσσα απαιτεί έναν αναλυτή μη τυπικού κειμένου που λογαριασμούς για χαρακτηριστικά από μια δεδομένη γλώσσα. Azure αναζήτησης προσφέρει δύο τύπους αναλυτές:

- αντίγραφα ασφαλείας των Lucene 35 αναλυτές.
- 50 αναλυτές αντίγραφα ασφαλείας των ΚΥΡΙΟΤΗΤΑΣ Microsoft φυσικής γλώσσας επεξεργασίας τεχνολογία που χρησιμοποιείται στο Office και το Bing.

Ορισμένοι προγραμματιστές ίσως προτιμάτε τη λύση πιο οικείο, απλή, Άνοιγμα προέλευσης του Lucene. Αναλυτές Lucene είναι πιο γρήγορα, αλλά το Microsoft αναλυτές έχουν προηγμένες δυνατότητες, όπως Λημματοποίηση, το word decompounding (σε γλώσσες όπως τα γερμανικά, δανικά, ολλανδικά, σουηδικά, νορβηγικά, εσθονικά, τέλος, ουγγρικά, Σλοβακικά) και αναγνώριση οντότητα (διευθύνσεις URL, μηνύματα ηλεκτρονικού ταχυδρομείου, ημερομηνίες, αριθμούς). Εάν είναι δυνατόν, θα πρέπει να εκτελέσετε συγκρίσεις από τη Microsoft και Lucene αναλυτές για να αποφασίσετε ποιο είναι ταιριάζουν καλύτερα.

***Πώς σύγκριση***

Η ανάλυση Lucene για τα Αγγλικά επεκτείνει την τυπική "Ανάλυση". Αυτό καταργεί Κτητικά (στο τέλος του) από λέξεις, ισχύει ρίζας σύμφωνα με [αλγόριθμο φέρουν ρίζας](http://tartarus.org/~martin/PorterStemmer/), και [Διακοπή λέξεων](http://en.wikipedia.org/wiki/Stop_words)στα Αγγλικά.

Με τη σύγκριση, Microsoft analyzer εκτελεί Λημματοποίηση αντί για ρίζας. Αυτό σημαίνει ότι μπορεί να χειριστεί φόρμες εσφαλμένες και ακανόνιστα word πολύ καλύτερα τι αποτελέσματα σε πιο συναφή αποτελέσματα αναζήτησης (δείτε ενότητα 7 της [παρουσίασης Azure αναζήτησης MVA](http://www.microsoftvirtualacademy.com/training-courses/adding-microsoft-azure-search-to-your-websites-and-apps) για περισσότερες λεπτομέρειες).

Δημιουργία ευρετηρίου με αναλυτές Microsoft είναι κατά μέσο όρο πιο αργά δύο έως τρεις φορές από τα ισοδύναμα Lucene, ανάλογα με τη γλώσσα. Απόδοση της αναζήτησης δεν πρέπει να επηρεαστούν σημαντικά για ερωτήματα μέσο μέγεθος.

***Ρύθμιση παραμέτρων***

Για κάθε πεδίο στον ορισμό ευρετήριο, μπορείτε να ορίσετε το `analyzer` ιδιότητας για ένα όνομα ανάλυσης που καθορίζει ποιες γλώσσας και προμηθευτή. Η ίδια ανάλυση θα εφαρμοστεί κατά τη δημιουργία ευρετηρίου και αναζήτηση για αυτό το πεδίο.
Για παράδειγμα, μπορείτε να έχετε ξεχωριστά πεδία για Αγγλικά, γαλλικά και τα Ισπανικά περιγραφές ξενοδοχείο που υπάρχουν δίπλα-δίπλα στο ίδιο ευρετήριο. Χρησιμοποιήστε την [παράμετρο ερωτήματος 'searchFields'](#SearchQueryParameters) για να καθορίσετε ποιο πεδίο συγκεκριμένη γλώσσα για να πραγματοποιήσετε αναζήτηση σε σχέση με στα ερωτήματά σας. Μπορείτε να δείτε παραδείγματα ερωτημάτων που περιλαμβάνουν το `analyzer` ιδιότητα στην [Αναζήτηση εγγράφων](#SearchDocs).

**Suggesters**

A `suggester` ορίζει ποια πεδία σε ένα ευρετήριο που χρησιμοποιούνται για την υποστήριξη αυτόματης καταχώρησης στις αναζητήσεις. Συνήθως συμβολοσειρές μερικής αναζήτησης στην οποία αποστέλλονται το [API προτάσεις](#Suggestions) ενώ πληκτρολογείτε ένα ερώτημα αναζήτησης και το API επιστρέφει ένα σύνολο προτεινόμενες φράσεις. Μια suggester που έχετε καθορίσει στο ευρετήριο προσδιορίζει ποια πεδία που χρησιμοποιούνται για τη δημιουργία τους όρους αναζήτησης πρόβλεψης. Για λεπτομέρειες ρύθμισης παραμέτρων, ανατρέξτε στο θέμα [Suggesters](#Suggesters) .

**Βαθμολογίας προφίλ**

A `scoringProfile` ορίζει προσαρμοσμένο βαθμολογίας συμπεριφορές που σας επιτρέπουν να επηρεάσετε ποια στοιχεία εμφανίζονται υψηλότερα από τα αποτελέσματα αναζήτησης. Προφίλ βαθμολογίας αποτελούνται από τις συναρτήσεις και διαφορετικό πάχος πεδίο. Για να χρησιμοποιήσετε τους, μπορείτε να καθορίσετε ένα προφίλ κατά όνομα στη συμβολοσειρά ερώτημα.

Μια προεπιλεγμένη βαθμολογίας προφίλ λειτουργεί στο παρασκήνιο για να υπολογίσετε μια βαθμολογία αναζήτησης για κάθε στοιχείο σε ένα σύνολο αποτελεσμάτων. Μπορείτε να χρησιμοποιήσετε το εσωτερικό, ανώνυμες βαθμολογίας προφίλ. Εναλλακτικά, ορίστε `defaultScoringProfile` για να χρησιμοποιήσετε ένα προσαρμοσμένο προφίλ ως την προεπιλεγμένη, ενεργοποιείται κάθε φορά που ένα προσαρμοσμένο προφίλ δεν έχει καθοριστεί σε συμβολοσειρά ερωτήματος.

Για λεπτομέρειες, ανατρέξτε στο θέμα [Προσθήκη βαθμολογίας προφίλ ένα ευρετήριο αναζήτησης (Azure Search REST API της υπηρεσίας)](search-api-scoring-profiles-2015-02-28-preview.md) .

**Επιλογές CORS**

Javascript πλευρά του προγράμματος-πελάτη δεν είναι δυνατό να καλέστε οποιαδήποτε APIs από προεπιλογή, επειδή το πρόγραμμα περιήγησης θα αποτρέψει όλες τις αιτήσεις σταυρό origin. Ενεργοποίηση CORS (κοινή χρήση του σταυρό Origin πόρου), ορίζοντας την `corsOptions` χαρακτηριστικό για να επιτρέψετε σε σταυρό origin ερωτημάτων για το ευρετήριό σας. Σημείωση το ερώτημα μόνο APIs υποστήριξη CORS για λόγους ασφαλείας. Μπορείτε να ορίσετε τις ακόλουθες επιλογές για CORS:

- `allowedOrigins`(απαιτείται): Αυτή είναι μια λίστα με τις προελεύσεις που θα σας εκχωρηθεί πρόσβαση για το ευρετήριό σας. Αυτό σημαίνει ότι οποιαδήποτε κώδικα Javascript που σερβιρίστηκε από αυτές τις προελεύσεις θα επιτρέπεται ερώτημα για το ευρετήριό σας (αν υποθέσουμε ότι παρέχει το σωστό κλειδί API). Κάθε origin είναι συνήθως της φόρμας `protocol://fully-qualified-domain-name:port` Παρόλο που η θύρα συχνά παραλείπεται. Ανατρέξτε [σε αυτό το άρθρο](http://go.microsoft.com/fwlink/?LinkId=330822) για περισσότερες λεπτομέρειες.
 - Εάν θέλετε να επιτρέψετε την πρόσβαση σε όλες τις προελεύσεις, συμπεριλάβετε `*` ως ένα στοιχείο στο το `allowedOrigins` πίνακα. Λάβετε υπόψη ότι **αυτό δεν συνιστάται πρακτική εξάσκηση για τις υπηρεσίες αναζήτησης παραγωγής.** Ωστόσο, ίσως είναι χρήσιμο για ανάπτυξη ή σκοπούς εντοπισμού σφαλμάτων.
- `maxAgeInSeconds`(προαιρετικό): προγράμματα περιήγησης Χρησιμοποιήστε αυτήν την τιμή για να προσδιορίσετε τη διάρκεια (σε δευτερόλεπτα) σε cache CORS προκαταρκτικού αποκρίσεις. Πρέπει να είναι μια μη αρνητικός ακέραιος. Αυτή η τιμή τόσο μεγαλύτερη είναι, θα είναι η καλύτερη απόδοση, αλλά τόσο περισσότερος χρόνος θα χρειαστεί για να τεθούν σε ισχύ οι αλλαγές πολιτικής CORS. Εάν δεν έχει ρυθμιστεί, θα χρησιμοποιηθεί μια προεπιλεγμένη διάρκεια 5 λεπτών.

<a name="Suggesters"></a>
## <a name="suggesters"></a>Suggesters

Η δυνατότητα προτάσεις στο πλαίσιο Αναζήτηση Azure είναι μια δυνατότητα ερωτήματος πρόβλεψης ή αυτόματης καταχώρησης, παρέχοντας μια λίστα με πιθανές όρους αναζήτησης απάντηση εισόδων μερική συμβολοσειρά καταχωρηθούν σε ένα πλαίσιο αναζήτησης. Ίσως έχετε παρατηρήσει προτάσεις ερωτημάτων κατά τη χρήση εμπορική μηχανισμοί αναζήτησης: πληκτρολογώντας ".NET" στο Bing δημιουργεί μια λίστα με τους όρους για ".NET 4,5", ".NET Framework 3,5", κ.ο.κ. Κατά τη χρήση της υπηρεσίας Search REST API, εφαρμογή προτάσεις σε μια προσαρμοσμένη εφαρμογή Azure αναζήτησης απαιτεί τα εξής:

- Ενεργοποίηση προτάσεων, προσθέτοντας μια κατασκευή **suggester** του ευρετηρίου, παρέχοντας το όνομα της λειτουργίας αναζήτησης και μια λίστα πεδίων για την οποία πρόβλεψης ενεργοποιείται. Για παράδειγμα, εάν καθορίσετε "ΌνομαΠόλης" ως πεδίο προέλευσης, πληκτρολογώντας μια συμβολοσειρά μερικής αναζήτησης "Θαλάσσια" θα έχει ως αποτέλεσμα "Σηάτλ", "Σωσίβιας" και "Seatac" (τους τρεις τύπους είναι τα ονόματα πραγματική πόλη) παρέχεται ως προτάσεις ερωτημάτων στο χρήστη.

- Κλήση προτάσεις καλώντας το [API προτάσεις](#Suggestions) κώδικα της εφαρμογής σας. Συνήθως συμβολοσειρές μερικής αναζήτησης στην οποία αποστέλλονται στην υπηρεσία ενώ πληκτρολογείτε ένα ερώτημα αναζήτησης και αυτό το API επιστρέφει ένα σύνολο προτεινόμενες φράσεις.

Σε αυτό το άρθρο εξηγεί πώς μπορείτε να ρυθμίσετε τις παραμέτρους ενός **suggester**. Μπορείτε, επίσης, θα πρέπει να αναθεωρήσετε το [API προτάσεις](#Suggestions) για λεπτομέρειες σχετικά με τον τρόπο χρήσης μιας suggester.

**Χρήση**

`Suggesters`δημιουργούνται στο ευρετήριο και εργασίας καλύτερα όταν χρησιμοποιούνται για την πρόταση συγκεκριμένα έγγραφα και όχι ελεύθερη όρους ή τις φράσεις. Τα καλύτερα πεδία υποψηφίου είναι τίτλοι, τα ονόματα και άλλα σχετικά σύντομο φράσεις που μπορούν να προσδιορίσουν ενός στοιχείου. Λιγότερο αποτελεσματική είναι επαναλαμβανόμενες πεδία, όπως κατηγορίες και ετικέτες, ή πολύ μεγάλο πεδία, όπως τα πεδία περιγραφές ή σχόλια.

Ως μέρος του ορισμού ευρετήριο, μπορείτε να προσθέσετε ένα μεμονωμένο suggester για να το `suggesters` συλλογής. Ιδιότητες που προσδιορίζουν μια suggester περιλαμβάνουν τα εξής:

- `name`: Το όνομα του το suggester. Μπορείτε να χρησιμοποιήσετε το όνομα του το suggester κατά την κλήση του `suggest` API.
- `searchMode`: Η στρατηγική που χρησιμοποιείται για την αναζήτηση υποψηφίου φράσεις. Είναι η μόνη κατάστασης λειτουργίας υποστηρίζονται αυτήν τη στιγμή `analyzingInfixMatching`, που εκτελεί ευέλικτη αντίστοιχες φράσεων στην αρχή ή στο μέσο προτάσεων.
- `sourceFields`: Μια λίστα με ένα ή περισσότερα πεδία που αποτελούν την προέλευση του περιεχομένου για προτάσεις. Μόνο τα πεδία του τύπου `Edm.String` και `Collection(Edm.String)` μπορεί να είναι προελεύσεις για προτάσεις. Μπορεί να χρησιμοποιηθεί μόνο τα πεδία που δεν έχετε έναν αναλυτή γλώσσας προσαρμοσμένη ρύθμιση.

**Παράδειγμα suggester**

Μια suggester αποτελεί μέρος του ευρετηρίου. Μπορεί να υπάρχει μόνο μία suggester σε το `suggesters` συλλογής στην τρέχουσα έκδοση, μαζί με τη συλλογή πεδίων και `scoringProfiles`.

        {
          "name": "hotels",
          "fields": [
             . . .
           ],
          "suggesters": [
            {
            "name": "sg",
            "searchMode": "analyzingInfixMatching",
            "sourceFields: ["hotelName", "category"]
            }
          ],
          "scoringProfiles": [
             . . .
          ]
        }

> [AZURE.NOTE]  Εάν χρησιμοποιήσατε την έκδοση preview δημόσια Azure αναζήτησης, `suggesters` αντικαθιστά μια παλαιότερη δυαδική ιδιότητα (`"suggestions": false`) που υποστηρίζεται μόνο πρόθεμα προτάσεις για τις συμβολοσειρές σύντομο (3-25 χαρακτήρες). Το αντικατάστασης, `suggesters`, υποστηρίζει τελεστής αντιστοίχιση που εντοπίζει όρων που ταιριάζουν στην αρχή ή στο μέσο το περιεχόμενο του πεδίου, με καλύτερη ανοχή για τυχόν λάθη σε συμβολοσειρές αναζήτησης. Ξεκινώντας με την έκδοση γενικά διαθέσιμο, αυτό είναι τώρα την εφαρμογή μόνο από τις προτάσεις API. Το παλαιότερο `suggestions` ιδιότητα που παρουσιάστηκε στο `api-version=2014-07-31-Preview` εξακολουθούν να λειτουργούν σε αυτήν την έκδοση, αλλά δεν λειτουργεί στο το `2015-02-28` ή νεότερες εκδόσεις Azure αναζήτησης.

<a name="UpdateIndex"></a>
## <a name="update-index"></a>Ενημέρωση ευρετηρίου

Μπορείτε να ενημερώσετε ένα υπάρχον ευρετήριο μέσα σε αναζήτηση Azure χρησιμοποιώντας μια αίτηση HTTP ΤΟΠΟΘΈΤΗΣΗ. Οι ενημερώσεις μπορούν να περιλαμβάνουν Προσθήκη νέων πεδίων στο υπάρχον σχήμα, τροποποίηση επιλογές CORS και την τροποποίηση βαθμολογίας προφίλ. Για περισσότερες πληροφορίες, ανατρέξτε στο θέμα [Προσθήκη βαθμολογίας προφίλ](https://msdn.microsoft.com/library/azure/dn798928.aspx) . Μπορείτε να καθορίσετε το όνομα του ευρετηρίου για να ενημερώσετε το αίτημα URI:

    PUT https://[search service url]/indexes/[index name]?api-version=[api-version]
    Content-Type: application/json
    api-key: [admin key]

**Σημαντικό:** Υποστήριξη για ενημερώσεις σχήματος ευρετηρίου περιορίζεται σε εργασίες που δεν απαιτούν φτιάχνετε το ευρετήριο αναζήτησης. Οι ενημερώσεις σχήματος που απαιτείται εκ νέου δημιουργία ευρετηρίου, όπως η αλλαγή τύποι πεδίων, δεν υποστηρίζονται αυτήν τη στιγμή. Νέα πεδία μπορούν να προστεθούν οποιαδήποτε στιγμή, παρόλο που δεν είναι δυνατό να αλλάξει ή να διαγραφούν υπάρχοντα πεδία. Το ίδιο ισχύει για `suggesters`. Νέα πεδία μπορούν να προστεθούν σε μια suggester τη στιγμή προστίθενται τα πεδία, αλλά τα πεδία δεν είναι δυνατό να καταργηθούν από `suggesters` και υπάρχοντα πεδία δεν μπορούν να προστεθούν σε `suggesters`.

Όταν προσθέτετε ένα νέο πεδίο σε ένα ευρετήριο, όλα τα υπάρχοντα έγγραφα στο ευρετήριο θα έχει αυτόματα μια τιμή null για αυτό το πεδίο. Χωρίς επιπλέον χώρο αποθήκευσης θα καταναλωθεί μέχρι να τα νέα έγγραφα που προστίθενται στο ευρετήριο.

**Αίτηση**

HTTPS είναι απαραίτητη για όλες τις αιτήσεις εξυπηρέτησης. Η αίτηση **Ενημέρωση ευρετηρίου** έχει συνταχθεί με χρήση ΤΟΠΟΘΈΤΗΣΗ HTTP. Με την ΤΟΠΟΘΈΤΗΣΗ, το όνομα του ευρετηρίου αποτελεί τμήμα της διεύθυνσης URL. Εάν δεν υπάρχει το ευρετήριο, δημιουργείται. Εάν το ευρετήριο υπάρχει ήδη, ενημερώνεται για το νέο ορισμό.

Το όνομα του ευρετηρίου πρέπει να είναι πεζά Έναρξη με ένα γράμμα ή έναν αριθμό, έχετε καμία καθέτους ή τελείες και είναι μικρότερη από 128 χαρακτήρες. Μετά την εκκίνηση το όνομα του ευρετηρίου με ένα γράμμα ή έναν αριθμό, το υπόλοιπο του ονόματος να συμπεριλάβετε οποιοδήποτε γράμμα, αριθμός και παύλες, με την προϋπόθεση ότι δεν είναι διαδοχικά το παύλες.

`api-version=[string]`(απαιτείται). Η έκδοση προεπισκόπησης είναι `api-version=2015-02-28-Preview`. Για λεπτομέρειες και εναλλακτικές εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης.

- `Content-Type`: Υποχρεωτικό. Ορίσετε την επιλογή`application/json`
- `api-key`: Υποχρεωτικό. Το `api-key` χρησιμοποιείται για τον έλεγχο ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικά στην υπηρεσία σας. Πρέπει να περιλαμβάνει την αίτηση **Ενημέρωση ευρετηρίου** μιας `api-key` κεφαλίδα το κλειδί διαχείρισης (σε αντίθεση με έναν αριθμό-κλειδί ερωτήματος).

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

**Σύνταξη σώμα αίτησης**

Κατά την ενημέρωση ένα υπάρχον ευρετήριο, ο οργανισμός πρέπει να συμπεριλάβετε τον αρχικό ορισμό σχήματος, καθώς και τα νέα πεδία που προσθέτετε, καθώς και τα έχουν τροποποιηθεί βαθμολογίας προφίλ, suggesters CORS επιλογές και, εάν υπάρχει. Εάν δεν θέλετε να τροποποιήσετε το προφίλ βαθμολογίας και τις επιλογές CORS, πρέπει να συμπεριλάβετε τα πρωτότυπα από κατά την οποία δημιουργήθηκε το ευρετήριο. Σε γενικές γραμμές το μοτίβο καλύτερα να χρησιμοποιήσετε για ενημερώσεις είναι για να ανακτήσετε τον ορισμό ευρετηρίου με ένα GET, τροποποιήστε το και κατόπιν ενημερώσετε με ΤΟΠΟΘΈΤΗΣΗ.

Η σύνταξη σχήματος που χρησιμοποιούνται για τη δημιουργία ευρετηρίου αναπαράγεται εδώ για τη διευκόλυνσή σας. Για περισσότερες λεπτομέρειες, ανατρέξτε στο θέμα [Δημιουργία ευρετηρίου](#CreateIndex) .

Από προεπιλογή ο οργανισμός απάντησης θα είναι κενή. Ωστόσο, εάν το `Prefer` κεφαλίδα αίτησης έχει οριστεί σε `return=representation`, ο οργανισμός απόκριση θα περιέχει το JSON για τον ορισμό ευρετηρίου που έχουν ενημερωθεί. Σε αυτήν την περίπτωση, θα είναι ο κωδικός κατάστασης επιτυχίας "200 OK".

**Ενημέρωση ευρετηρίου ορισμού με προσαρμοσμένο αναλυτές**

Όταν μια ανάλυση, ενός αντικειμένου δημιουργίας διακριτικών, διακριτικού φίλτρο ή ένα φίλτρο char ορίζεται, δεν μπορεί να τροποποιηθεί. Νέα μπορούν να προστεθούν σε ένα υπάρχον ευρετήριο μόνο εάν η `allowIndexDowntime` σημαία έχει οριστεί στην τιμή true στην πρόσκληση σε ενημέρωση ευρετηρίου: 

`PUT https://[search service name].search.windows.net/indexes/[index name]?api-version=[api-version]&allowIndexDowntime=true`

Σημειώστε ότι αυτή η λειτουργία θα τοποθετήσετε το ευρετήριό σας εκτός σύνδεσης για τουλάχιστον μερικά δευτερόλεπτα, προκαλεί τη δημιουργία του ευρετηρίου και προσκλήσεις ερωτήματος αποτυχία. Διαθεσιμότητα επιδόσεις και την εγγραφή του ευρετηρίου μπορεί να είναι με προβλήματα όρασης για αρκετά λεπτά μετά την ενημέρωση του ευρετηρίου ή περισσότερο χρόνο για πολύ μεγάλες ευρετήρια.

<a name="DocOps"></a>
## <a name="document-operations"></a>Λειτουργίες εγγράφων

Στο πλαίσιο Αναζήτηση Azure, ένα ευρετήριο είναι αποθηκευμένο στο cloud και συμπληρωμένο με χρήση JSON εγγράφων που στέλνετε στην υπηρεσία. Όλα τα έγγραφα που κάνετε αποστολή περιλαμβάνει του σώματος των δεδομένων σας αναζήτησης. Τα έγγραφα περιέχουν πεδία, ορισμένες από τις οποίες είναι με διακριτικά σε όρους αναζήτησης κατά την αποστολή τους. Το `/docs` τμήματος διεύθυνση URL στο API αναζήτησης Azure αντιπροσωπεύει τη συλλογή εγγράφων σε ένα ευρετήριο. Όλες οι λειτουργίες που εκτελούνται στη συλλογή όπως αποστολή, τη συγχώνευση, διαγραφή ή υποβολή ερωτημάτων λήψη εγγράφων τοποθετήσετε στο περιβάλλον της ένα ευρετήριο, επομένως, τις διευθύνσεις URL για αυτές τις λειτουργίες θα ξεκινάτε πάντα με `/indexes/[index name]/docs` για ένα όνομα που δίνεται ευρετηρίου.

Κώδικα της εφαρμογής σας πρέπει να είτε για να δημιουργήσετε έγγραφα JSON για αποστολή στο Azure αναζήτησης ή μπορείτε να χρησιμοποιήσετε ένα [ευρετήριο](https://msdn.microsoft.com/library/dn946891.aspx) για να φορτώσετε έγγραφα, εάν το αρχείο προέλευσης δεδομένων είναι βάση δεδομένων SQL Azure ή DocumentDB. Συνήθως, ευρετήρια συμπληρώνονται από ένα μεμονωμένο σύνολο δεδομένων που παρέχετε.

Θα πρέπει να σκοπεύετε να αντιμετωπίζετε ένα έγγραφο για κάθε στοιχείο που θέλετε να πραγματοποιήσετε αναζήτηση. Μια εφαρμογή ενοικίασης ταινίας ενδέχεται να έχετε ένα έγγραφο ανά ταινίας, μια εφαρμογή ηλεκτρονικό κατάστημα μπορεί να έχετε ένα έγγραφο ανά SKU, μια εφαρμογή του εκπαιδευτικού λογισμικού online μπορεί να έχετε ένα έγγραφο ανά κύκλου, μια επιχείρηση έρευνα ενδέχεται να έχετε ένα έγγραφο για κάθε ακαδημαϊκό χαρτιού στο αποθετήριο τους και ούτω καθεξής.

Έγγραφα που αποτελείται από ένα ή περισσότερα πεδία. Πεδία μπορεί να περιέχει κείμενο που είναι με διακριτικά από την αναζήτηση Azure σε όρους αναζήτησης, καθώς και μη με διακριτικά ή τιμές χωρίς κείμενο που μπορεί να χρησιμοποιηθεί σε φίλτρα ή βαθμολογίας προφίλ. Τα ονόματα, τους τύπους δεδομένων και δυνατότητες αναζήτησης που υποστηρίζονται για κάθε πεδίο καθορίζονται από το σχήμα ευρετηρίου. Ένα από τα πεδία σε κάθε διάταξη ευρετήριο πρέπει να θεωρούνται ως Αναγνωριστικό και κάθε έγγραφο πρέπει να έχει μια τιμή για το πεδίο Αναγνωριστικό που προσδιορίζει με μοναδικό τρόπο αυτό το έγγραφο στο ευρετήριο. Όλα τα άλλα πεδία εγγράφου είναι προαιρετικές και προεπιλεγμένη τιμή null Εάν δεν καθορίζεται αριστερά. Σημειώστε ότι οι τιμές null δεν καταλαμβάνουν χώρο στο ευρετήριο αναζήτησης.

Πριν να αποστείλετε έγγραφα, πρέπει να έχετε ήδη δημιουργήσει το ευρετήριο στην υπηρεσία. Για λεπτομέρειες σχετικά με αυτό το πρώτο βήμα, ανατρέξτε στο θέμα [Δημιουργία ευρετηρίου](#CreateIndex) .

<a name="AddOrUpdateDocuments"></a>
## <a name="add-update-or-delete-documents"></a>Προσθήκη, ενημέρωση ή διαγραφή εγγράφων

Μπορείτε να αποστείλετε, συγχώνευση, συγχώνευση ή αποστολή ή διαγραφή εγγράφων από ένα καθορισμένο ευρετήριο χρησιμοποιώντας HTTP POST. Για μεγάλο αριθμό ενημερώσεις, συνιστάται η δέσμη εγγράφων έως (1000 έγγραφα ανά δέσμη) ή περίπου 16 MB ανά δέσμης.

    POST https://[service name].search.windows.net/indexes/[index name]/docs/index?api-version=[api-version]
    Content-Type: application/json
    api-key: [admin key]

**Αίτηση**

HTTPS είναι απαραίτητη για όλες τις αιτήσεις εξυπηρέτησης. Μπορείτε να αποστείλετε, συγχώνευση, συγχώνευση ή αποστολή ή διαγραφή εγγράφων από ένα καθορισμένο ευρετήριο χρησιμοποιώντας HTTP POST.

Η αίτηση URI περιλαμβάνει [όνομα ευρετηρίου], που καθορίζει ποιες ευρετηρίου για να δημοσιεύσετε έγγραφα. Μπορείτε μόνο να δημοσιεύσετε έγγραφα σε ένα ευρετήριο κάθε φορά.

`api-version=[string]`(απαιτείται). Η έκδοση προεπισκόπησης είναι `api-version=2015-02-28-Preview`. Για λεπτομέρειες και εναλλακτικές εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης.

- `Content-Type`: Υποχρεωτικό. Ορίσετε την επιλογή`application/json`
- `api-key`: Υποχρεωτικό. Το `api-key` χρησιμοποιείται για τον έλεγχο ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικά στην υπηρεσία σας. Πρέπει να περιλαμβάνει την πρόσκληση **Προσθήκη εγγράφων** σε ένα `api-key` κεφαλίδα το κλειδί διαχείρισης (σε αντίθεση με έναν αριθμό-κλειδί ερωτήματος).

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

**Σώμα αίτησης**

Στο σώμα της πρόσκλησης σε περιέχει ένα ή περισσότερα έγγραφα να τοποθετηθούν στο ευρετήριο. Έγγραφα αναγνωρίζονται από ένα μοναδικό κλειδί. Κάθε έγγραφο είναι συσχετισμένη με μια ενέργεια: Αποστολή, συγχώνευση, mergeOrUpload ή διαγραφή. Αποστολή προσκλήσεων πρέπει να περιλαμβάνουν τα δεδομένα του εγγράφου ως σύνολο ζεύγη κλειδιού/τιμής.

    {
      "value": [
        {
          "@search.action": "upload (default) | merge | mergeOrUpload | delete",
          "key_field_name": "unique_key_of_document", (key/value pair for key field from index schema)
          "field_name": field_value (key/value pairs matching index schema)
            ...
        },
        ...
      ]
    }

> [AZURE.NOTE] Πλήκτρα εγγράφου μπορεί να περιέχει μόνο γράμματα, αριθμούς, παύλες ("-"), χαρακτήρες υπογράμμισης ("_") και το σύμβολο ίσον ("="). Για περισσότερες λεπτομέρειες, ανατρέξτε στο άρθρο [κανόνες ονομάτων](https://msdn.microsoft.com/library/azure/dn857353.aspx).

**Ενέργειες εγγράφου**

- `upload`: Μια ενέργεια αποστολής είναι παρόμοια με μια "upsert" όπου θα εισαχθεί το έγγραφο εάν πρόκειται για νέα και ενημέρωση/αντικατασταθεί εάν υπάρχει. Σημειώστε ότι όλα τα πεδία έχουν αντικατασταθεί στην περίπτωση που η ενημερωμένη έκδοση.
- `merge`: Συγχώνευση ενημερώνεται ένα υπάρχον έγγραφο με το καθορισμένο πεδία. Εάν δεν υπάρχει το έγγραφο, θα αποτύχει η συγχώνευση. Οποιοδήποτε πεδίο που καθορίζετε σε μια συγχώνευση, θα αντικαταστήσει το υπάρχον πεδίο στο έγγραφο. Αυτό περιλαμβάνει πεδία του τύπου `Collection(Edm.String)`. Για παράδειγμα, εάν το έγγραφο περιέχει ένα πεδίο "ετικέτες" με τιμή `["budget"]` και την εκτέλεση μιας συγχώνευσης με τιμή `["economy", "pool"]` για "ετικέτες", θα είναι η τελική τιμή του πεδίου "ετικέτες" `["economy", "pool"]`. Αυτό θα **δεν** είναι `["budget", "economy", "pool"]`.
- `mergeOrUpload`: συμπεριφέρεται σαν `merge` εάν υπάρχει ήδη ένα έγγραφο με τον αριθμό-κλειδί που δίνεται στο ευρετήριο. Εάν το έγγραφο δεν υπάρχει το συμπεριφέρεται όπως `upload` με ένα νέο έγγραφο.
- `delete`: Διαγραφή καταργεί το καθορισμένο έγγραφο από το ευρετήριο. Σημειώστε ότι τα πεδία που καθορίζετε σε ένα `delete` λειτουργία εκτός από το πεδίο κλειδιού θα αγνοούνται. Εάν θέλετε να καταργήσετε ένα μεμονωμένο πεδίο από ένα έγγραφο, χρησιμοποιήστε `merge` απλώς και αντί για αυτό, ορίστε ρητά για το πεδίο `null`.

**Απόκριση**

Ο κωδικός κατάστασης 200 (OK) επιστρέφεται για μια επιτυχημένη απόκριση, γεγονός που σημαίνει ότι όλα τα στοιχεία έχουν καταχωρηθεί με επιτυχία στο ευρετήριο. Αυτό υποδεικνύεται από το `status` την ιδιότητα οριστεί σε true για όλα τα στοιχεία, καθώς και ως το `statusCode` ιδιότητα που ορίζεται σε 201 (για τα έγγραφα που μόλις έχει αποσταλεί) ή 200 (για έγγραφα συγχωνευμένα ή διαγραφεί):

    {
      "value": [
        {
          "key": "unique_key_of_new_document",
          "status": true,
          "errorMessage": null,
          "statusCode": 201
        },
        {
          "key": "unique_key_of_merged_document",
          "status": true,
          "errorMessage": null,
          "statusCode": 200
        },
        {
          "key": "unique_key_of_deleted_document",
          "status": true,
          "errorMessage": null,
          "statusCode": 200
        }
      ]
    }  

Ο κωδικός κατάστασης 207 (πολλών κατάστασης) επιστρέφεται όταν τουλάχιστον ένα στοιχείο δεν καταχωρήθηκε με επιτυχία στο ευρετήριο. Στοιχεία που δεν έχουν καταχωρηθεί στο ευρετήριο έχουν το `status` πεδίο την τιμή false. Το `errorMessage` και `statusCode` ιδιότητες θα υποδεικνύουν το λόγο του ευρετηρίου σφάλματος:

    {
      "value": [
        {
          "key": "unique_key_of_document_1",

<a name="SearchDocs"></a>
## <a name="search-documents"></a>Αναζήτηση εγγράφων

Μια λειτουργία **αναζήτησης** εκδίδεται ως μια αίτηση GET ή ΔΗΜΟΣΊΕΥΣΗ και καθορίζει τις παραμέτρους που παρέχουν τα κριτήρια για την επιλογή που ταιριάζει έγγραφα.

    GET https://[service name].search.windows.net/indexes/[index name]/docs?[query parameters]
    api-key: [admin or query key]

    POST https://[service name].search.windows.net/indexes/[index name]/docs/search?api-version=[api-version]
    Content-Type: application/json
    api-key: [admin or query key]

**Πότε να χρησιμοποιήσετε ΔΗΜΟΣΊΕΥΣΗ αντί για ΛΉΨΗ**

Όταν χρησιμοποιείτε HTTP GET για να καλέσετε την **Αναζήτηση** API, πρέπει να έχετε υπόψη ότι το μήκος της διεύθυνσης URL της αίτησης δεν πρέπει να υπερβαίνει 8 KB. Αυτό είναι αρκετά συνήθως για τις περισσότερες εφαρμογές. Ωστόσο, ορισμένες εφαρμογές παράγουν πολύ μεγάλες ερωτήματα ή παραστάσεις φίλτρου OData. Για αυτές τις εφαρμογές, χρησιμοποιώντας HTTP POST είναι καλύτερη επιλογή επειδή επιτρέπει μεγαλύτερο φίλτρα και ερωτήματα από ΛΉΨΗ. Με την ΚΑΤΑΧΏΡΗΣΗ, τον αριθμό των όρων ή όρων σε ένα ερώτημα είναι ο περιορισμός παράγοντας, όχι το μέγεθος του ερωτήματος ανεπεξέργαστα εφόσον το όριο μεγέθους αίτηση για ΔΗΜΟΣΊΕΥΣΗ είναι περίπου 16 MB.

> [AZURE.NOTE] Παρόλο που το όριο μεγέθους αίτηση ΔΗΜΟΣΊΕΥΣΗ είναι πολύ μεγάλο, δεν μπορεί να είναι αυθαίρετα σύνθετα ερωτήματα αναζήτησης και παραστάσεις φίλτρου. Ανατρέξτε στο θέμα [Lucene σύνταξη του ερωτήματος](https://msdn.microsoft.com/library/mt589323.aspx) και τη [σύνταξη παραστάσεων OData](https://msdn.microsoft.com/library/dn798921.aspx) για περισσότερες πληροφορίες σχετικά με περιορισμούς πολυπλοκότητα query και το φίλτρο αναζήτησης.

**Αίτηση**

HTTPS είναι απαραίτητη για αιτήσεις εξυπηρέτησης. Η αίτηση **αναζήτησης** μπορεί να κατασκευαστεί με τις μεθόδους GET ή ΔΗΜΟΣΊΕΥΣΗ.

Η αίτηση URI καθορίζει ποιο ευρετήριο σε ερώτημα, για όλα τα έγγραφα που ταιριάζουν με τις παραμέτρους. Παράμετροι έχουν καθοριστεί στη συμβολοσειρά ερώτημα στην περίπτωση αιτήσεις ΛΉΨΗΣ και, στην πρόσκληση σε προσκλήσεις σώμα στην περίπτωση ΚΑΤΑΧΏΡΗΣΗΣ.

Ως βέλτιστη πρακτική κατά τη δημιουργία αιτήσεις ΛΉΨΗΣ, μην ξεχάσετε να [κωδικοποίηση URL](https://msdn.microsoft.com/library/system.uri.escapedatastring.aspx) συγκεκριμένο ερώτημα παραμέτρων, όταν καλείτε το REST API απευθείας. Για λειτουργίες **αναζήτησης** , περιλαμβάνει:

- `$filter`
- `facet`
- `highlightPreTag`
- `highlightPostTag`
- `search`
- `moreLikeThis`

Κωδικοποίηση URL συνιστάται μόνο για τις παραπάνω παραμέτρους ερωτήματος. Εάν που κατά λάθος κωδικοποίηση URL τη συμβολοσειρά ολόκληρου του ερωτήματος (όλα τα στοιχεία μετά το;), θα διακόψει αιτήσεις.

Επίσης, κωδικοποίηση URL απαιτείται μόνο όταν καλείτε το REST API απευθείας με ΛΉΨΗ. Δεν υπάρχει κωδικοποίηση URL είναι απαραίτητο κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ ή όταν χρησιμοποιείτε το [.NET βιβλιοθήκη προγράμματος-πελάτη](https://msdn.microsoft.com/library/dn951165.aspx), η οποία χειρίζεται κωδικοποίηση URL για εσάς.

<a name="SearchQueryParameters"></a>
**Παράμετροι ερωτήματος**

**Αναζήτηση** αποδέχεται πολλούς παραμέτρους που παρέχουν κριτήρια ερωτήματος και επίσης να καθορίσετε τη συμπεριφορά αναζήτησης. Μπορείτε να παράσχετε αυτές τις παραμέτρους στη διεύθυνση URL κατά την κλήση **αναζήτησης** μέσω ΛΉΨΗ και ως JSON ιδιότητες στο σώμα της αίτησης κατά την κλήση **αναζήτησης** μέσω ΔΗΜΟΣΊΕΥΣΗ συμβολοσειρά ερωτήματος. Η σύνταξη για ορισμένες παράμετροι είναι λίγο διαφορετικά μεταξύ ΛΉΨΗ και ΔΗΜΟΣΊΕΥΣΗ. Οι διαφορές αυτές επισημαίνονται ανάλογα με την περίπτωση παρακάτω:

`search=[string]`(προαιρετικό) - το κείμενο για την αναζήτηση. Όλα `searchable` πεδία γίνεται αναζήτηση από προεπιλογή, εκτός εάν `searchFields` έχει καθοριστεί. Κατά την αναζήτηση `searchable` πεδία, το ίδιο το κείμενο αναζήτησης είναι με διακριτικά, ώστε να πολλούς όρους διαχωρίζονται με κενό διάστημα (για παράδειγμα: `search=hello world`). Για να ταιριάζει με οποιοδήποτε όρο, χρησιμοποιήστε `*` (αυτό είναι χρήσιμο για ερωτήματα boolean φίλτρο). Παράλειψη αυτή η παράμετρος έχει το ίδιο αποτέλεσμα με τη ρύθμιση του για να `*`. Για λεπτομέρειες σχετικά με τη σύνταξη αναζήτησης, ανατρέξτε στο θέμα [Απλό σύνταξη του ερωτήματος](https://msdn.microsoft.com/library/dn798920.aspx) .

  - **Σημείωση**: τα αποτελέσματα μερικές φορές μπορεί να είναι ήταν αναμενόμενο κατά την υποβολή ερωτήματος μέσω `searchable` πεδία. Το αντικειμένου δημιουργίας διακριτικών περιλαμβάνει λογικής να χειρίζεται τις περιπτώσεις κοινές σε κείμενο στα Αγγλικά όπως αποστρόφους, Χρήση κομμάτων σε αριθμούς, κ.λπ. Για παράδειγμα, `search=123,456` θα ταιριάζουν με ενός μεμονωμένου όρου 123,456 και όχι τα μεμονωμένα όρους 123 και 456, επειδή χρησιμοποιούνται κόμματα ως διαχωριστικά χιλιάδων για μεγάλου αριθμού στα Αγγλικά. Για αυτόν το λόγο, συνιστάται να χρησιμοποιείτε κενά διαστήματα αντί για σημεία στίξης για το διαχωρισμό των όρων στα το `search` παραμέτρου.

`searchMode=any|all`(προαιρετικά, από προεπιλογή `any`) - Εάν κάποιο ή όλα από τους όρους αναζήτησης πρέπει να αντιστοιχιστούν με για να μετρήσετε το έγγραφο ως συμφωνία.

`searchFields=[string]`(προαιρετικό) - τη λίστα των ονομάτων πεδίων διαχωρισμένες με κόμματα, για να αναζητήσετε το καθορισμένο κείμενο. Τα πεδία προορισμού πρέπει να επισημανθούν ως `searchable`.

`queryType=simple|full`(προαιρετικά, από προεπιλογή `simple`) - όταν ρυθμιστεί το κείμενο "απλή" Αναζήτηση ερμηνεύεται χρησιμοποιώντας μια γλώσσα απλού ερωτήματος που επιτρέπει για τα σύμβολα, όπως +, * και "". Ερωτήματα που αξιολογούνται σε όλα τα πεδία με δυνατότητα αναζήτησης (ή πεδία που υποδεικνύεται στην `searchFields`) σε κάθε έγγραφο από προεπιλογή. Όταν ο τύπος ερωτήματος έχει οριστεί σε `full` ερμηνεύεται αναζήτηση κειμένου με χρήση της γλώσσας ερωτήματος Lucene που επιτρέπει σε συγκεκριμένο πεδίο και σταθμισμένου αναζητήσεις. Ανατρέξτε στο θέμα [Απλό σύνταξη του ερωτήματος](https://msdn.microsoft.com/library/dn798920.aspx) και [Σύνταξη του ερωτήματος Lucene](https://msdn.microsoft.com/library/mt589323.aspx) για συγκεκριμένες απαιτήσεις σε το συντάξεις αναζήτησης. 
 
> [AZURE.NOTE] Περιοχή αναζήτησης στο το Lucene γλώσσα ερωτημάτων δεν υποστηρίζεται και θα προτιμήσει $filter οποίο προσφέρει παρόμοια λειτουργικότητα.

`moreLikeThis=[key]`(προαιρετικό) **Σημαντικό:** Αυτή η δυνατότητα είναι διαθέσιμη μόνο σε `2015-02-28-Preview`. Αυτή η επιλογή δεν μπορεί να χρησιμοποιηθεί σε ένα ερώτημα που περιέχει την παράμετρο αναζήτηση κειμένου, `search=[string]`. Το `moreLikeThis` παραμέτρου εντοπίζει εγγράφων που είναι παρόμοιο με το έγγραφο που καθορίζεται από τον αριθμό-κλειδί εγγράφου. Όταν γίνεται μια αίτηση αναζήτησης με `moreLikeThis`, μια λίστα με τους όρους αναζήτησης δημιουργείται με βάση τη συχνότητα και rarity όρων στο έγγραφο προέλευσης. Αυτούς τους όρους που χρησιμοποιούνται, στη συνέχεια, να κάνετε την αίτηση. Από προεπιλογή, τα περιεχόμενα όλων `searchable` θεωρείται ότι τα πεδία, εκτός εάν `searchFields` χρησιμοποιείται για να περιορίσετε τα πεδία στα οποία γίνεται αναζήτηση.  

`$skip=#`(προαιρετικό) - τον αριθμό των αποτελεσμάτων αναζήτησης για να μεταβείτε γρήγορα; Δεν είναι μεγαλύτερο από 100,000. Εάν πρέπει να εκτελέσετε σάρωση εγγράφων με τη σειρά, αλλά δεν μπορείτε να χρησιμοποιήσετε `$skip` οφείλεται σε αυτόν τον περιορισμό, εξετάστε το ενδεχόμενο χρήσης `$orderby` σε έναν αριθμό-κλειδί συνολικά παραγγελίας και `$filter` με μια περιοχή ερωτήματος αντί για αυτό.

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `skip` αντί για `$skip`.

`$top=#`(προαιρετικό) - τον αριθμό των αποτελεσμάτων αναζήτησης για να ανακτήσετε. Αυτό μπορεί να χρησιμοποιηθεί σε συνδυασμό με `$skip` για την εφαρμογή-πελάτη σελιδοποίησης των αποτελεσμάτων αναζήτησης.

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `top` αντί για `$top`.

`$count=true|false`(προαιρετικά, από προεπιλογή `false`)-καθορίζει αν θα γίνεται λήψη το συνολικό πλήθος των αποτελεσμάτων. Αυτό είναι το πλήθος όλων των εγγράφων που ταιριάζουν με το `search` και `$filter` παραμέτρους, παραβλέποντας `$top` και `$skip`. Επιλογή της τιμής `true` ενδέχεται να έχουν επίδραση επιδόσεων. Σημειώστε ότι η μέτρηση που επιστρέφεται είναι μια προσέγγιση.

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `count` αντί για `$count`.

`$orderby=[string]`(προαιρετικό) - μια λίστα από παραστάσεις διαχωρισμένες με κόμματα, για να ταξινομήσετε τα αποτελέσματα από. Κάθε παράσταση μπορεί να είναι είτε ένα όνομα πεδίου ή μια κλήση για να το `geo.distance()` συνάρτηση. Κάθε παράσταση μπορεί να ακολουθείται από `asc` για να υποδεικνύεται αύξουσα, και `desc` για να υποδείξετε φθίνουσα σειρά. Η προεπιλογή είναι αύξουσα σειρά. Από τις βαθμολογίες αντιστοιχία με τα έγγραφα που θα διαχωριστεί TIES. Εάν δεν υπάρχει `$orderby` έχει καθοριστεί, η προεπιλεγμένη σειρά ταξινόμησης φθίνουσα ταξινόμηση βάσει βαθμολογία match εγγράφου. Υπάρχει ένα όριο 32 όρων για `$orderby`.

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `orderby` αντί για `$orderby`.

`$select=[string]`(προαιρετικό) - μια λίστα πεδίων διαχωρισμένες με κόμματα, για να ανακτήσετε. Εάν δεν καθορίζεται, περιλαμβάνονται όλα τα πεδία που έχει επισημανθεί ως ανακτήσιμη στο σχήμα. Μπορείτε να ζητήσετε επίσης ρητά όλα τα πεδία, ορίζοντας αυτή η παράμετρος σε `*`.

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `select` αντί για `$select`.

`facet=[string]`(κανέναν ή περισσότερους) - ενός πεδίου για την όψη από. Προαιρετικά, η συμβολοσειρά μπορεί να περιέχει τις παραμέτρους για να προσαρμόσετε το faceting εκφρασμένη διαχωρισμένες με κόμματα `name:value` ζεύγη. Έγκυρες παράμετροι είναι οι εξής:

- `count`(μέγιστο αριθμό όρων όψη, η προεπιλογή είναι 10). Δεν υπάρχει καμία μέγιστη, αλλά υψηλότερες τιμές αναλαμβάνουν μια αντίστοιχη μειονέκτημα στις επιδόσεις, ειδικά εάν το πεδίο με συνθήκη περιέχει μεγάλο αριθμό μοναδικών όρους.
  - Για παράδειγμα: `facet=category,count:5` λαμβάνει τις κατηγορίες πέντε στα αποτελέσματα όψη.  
  - **Σημείωση**: Εάν το `count` παραμέτρου είναι μικρότερος από τον αριθμό των μοναδικών όρους, τα αποτελέσματα ενδέχεται να μην είναι ακριβείς. Αυτό οφείλεται όπως κατανέμονται faceting ερωτήματα σε shards. Αύξηση `count` γενικά αυξάνει την ακρίβεια της τις μετρήσεις όρου, αλλά με κόστος επιδόσεων.
- `sort`(μία από `count` για να ταξινομήσετε με *Φθίνουσα σειρά* από απογραφή, `-count` για να ταξινομήσετε *κατά αύξουσα* από απογραφή, `value` για να ταξινομήσετε *κατά αύξουσα* κατά τιμή, ή `-value` για να ταξινομήσετε με *Φθίνουσα σειρά* κατά τιμή)
  - Για παράδειγμα: `facet=category,count:3,sort:count` λαμβάνει τις κατηγορίες αρχή τριών στα αποτελέσματα όψη κατά φθίνουσα σειρά, ο αριθμός των εγγράφων με κάθε όνομα της πόλης. Για παράδειγμα, εάν επάνω τρεις κατηγορίες είναι προϋπολογισμού, Motel και πολυτελείας, και προϋπολογισμού περιλαμβάνει 5 επισκέψεων, Motel έχει 6 και πολυτελείας έχει 4, στη συνέχεια, το Κάδοι θα με τη σειρά Motel, ο προϋπολογισμός, πολυτελείας.
  - Για παράδειγμα: `facet=rating,sort:-value` παράγει Κάδοι για όλοι οι πιθανές χαρακτηρισμοί, σε φθίνουσα σειρά κατά τιμή. Για παράδειγμα, εάν είναι τους χαρακτηρισμούς από 1 έως 5, το Κάδοι θα παραγγελθούν 5, 4, 3, 2, 1, ανεξάρτητα από τον αριθμό των εγγράφων ταιριάζει με κάθε χαρακτηρισμό.
- `values`(Διαχ.: διοχέτευση αριθμητικό ή `Edm.DateTimeOffset` τιμές που καθορίζει ένα δυναμικό σύνολο τιμών καταχώρησης όψη)
  - Για παράδειγμα: `facet=baseRate,values:10|20` παράγει τρεις Κάδοι: μία για ποσό βάσης 0 έως αλλά εκτός της έως 10, μία για 10 αλλά όχι τα 20 και μία για 20 ή νεότερη έκδοση.
  - Για παράδειγμα: `facet=lastRenovationDate,values:2010-02-01T00:00:00Z` δημιουργεί δύο Κάδοι: μία για ξενοδοχεία ανακαινιζόμενα πριν Φεβρουαρίου 2010 και μία για ξενοδοχεία ανακαινιζόμενα Φεβρουαρίου 1ος, 2010 ή νεότερη έκδοση.
- `interval`(διάστημα ακέραιος αριθμός μεγαλύτερος του 0 για τους αριθμούς, ή `minute`, `hour`, `day`, `week`, `month`, `quarter`, `year` για τιμές ημερομηνίας ώρα)
  - Για παράδειγμα: `facet=baseRate,interval:100` παράγει Κάδοι που βασίζονται σε περιοχές βάσης επιτόκιο του μεγέθους 100. Για παράδειγμα, εάν βάσης συντελεστές είναι όλα μεταξύ $60 και $600, θα Κάδοι για 0-100, 100-200, 200 300, 300-400, 400-500 και 500-600.
  - Για παράδειγμα: `facet=lastRenovationDate,interval:year` δημιουργεί μία χρωματισμού για κάθε έτος όταν ξενοδοχεία έχουν ανακαινιζόμενα.
- `timeoffset`([+-] ωω: λλ, [+-] ωω: λλ, ή [+-] hh) `timeoffset` είναι προαιρετικό. Μόνο μπορεί να συνδυαστεί με το `interval` επιλογής, και μόνο όταν εφαρμόζονται σε ένα πεδίο τύπου `Edm.DateTimeOffset`. Η τιμή καθορίζει την απόκλιση UTC ώρα με το λογαριασμό για κατά τη ρύθμιση ώρας όρια.
  - Για παράδειγμα: `facet=lastRenovationDate,interval:day,timeoffset:-01:00` χρησιμοποιεί το όριο ημέρα που ξεκινά στο UTC 01:00:00 (μεσάνυχτα στη ζώνη ώρας προορισμού)
- **Σημείωση**: `count` και `sort` μπορεί να συνδυαστεί στην ίδια προδιαγραφή όψη, αλλά αυτές δεν μπορεί να συνδυαστεί με `interval` ή `values`, και `interval` και `values` δεν μπορεί να συνδυαστεί μαζί.
- **Σημείωση**: όψεις διαστήματος στην ημερομηνία ώρα υπολογίζονται που βασίζονται σε ώρα UTC εάν `timeoffset` δεν έχει καθοριστεί. Για παράδειγμα: για `facet=lastRenovationDate,interval:day`, της ημέρας όριο αρχίζει 00:00:00 UTC. 

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `facets` αντί για `facet`. Επίσης, μπορείτε το καθορίσετε ως πίνακας JSON συμβολοσειρών όπου κάθε συμβολοσειρά είναι μια ξεχωριστή όψη παράσταση.

`$filter=[string]`(προαιρετικό) - μια παράσταση δομημένες αναζήτησης σε τυπική σύνταξη OData. Για λεπτομέρειες σχετικά με το υποσύνολο των τη γραμματική παράσταση OData που υποστηρίζει το Azure αναζήτησης, ανατρέξτε στο θέμα [Σύνταξη παραστάσεων OData](#ODataExpressionSyntax) .

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `filter` αντί για `$filter`.

`highlight=[string]`(προαιρετικό) - οι επισημαίνει ένα σύνολο ονομάτων πεδίων διαχωρισμένες με κόμματα που χρησιμοποιούνται για την ακρίβεια. Μόνο `searchable` πεδία μπορούν να χρησιμοποιηθούν για την επισήμανση επισκέψεων.

`highlightPreTag=[string]`(προαιρετικά, από προεπιλογή `<em>`) - μια ετικέτα που τοποθετεί μπροστά από για να πατήσετε βασικά σημεία της συμβολοσειράς. Πρέπει να οριστεί με `highlightPostTag`.

> [AZURE.NOTE] Όταν καλέσετε **αναζήτησης** με χρήση GET, δεσμευμένο χαρακτήρες στη διεύθυνση URL πρέπει να είναι τοις εκατό κωδικοποιημένη (για παράδειγμα, αντί για το # % 23).

`highlightPostTag=[string]`(προαιρετικά, από προεπιλογή `</em>`)-μια ετικέτα συμβολοσειρά που προσαρτά να το πετύχετε επισημάνσεις. Πρέπει να οριστεί με `highlightPreTag`.

> [AZURE.NOTE] Όταν καλέσετε **αναζήτησης** με χρήση GET, δεσμευμένο χαρακτήρες στη διεύθυνση URL πρέπει να είναι τοις εκατό κωδικοποιημένη (για παράδειγμα, αντί για το # % 23).

`scoringProfile=[string]`(προαιρετικό) - στο όνομα της βαθμολογίας προφίλ για την αξιολόγηση συμφωνεί με βαθμολογίες αντίστοιχων έγγραφα για να ταξινομήσετε τα αποτελέσματα.

`scoringParameter=[string]`(κανέναν ή περισσότερους) - υποδεικνύει τις τιμές για κάθε παράμετρο που έχει οριστεί σε μια συνάρτηση βαθμολογίας (για παράδειγμα, `referencePointParameter`) χρησιμοποιώντας τη μορφή `name-value1,value2,...`.

- Για παράδειγμα, εάν το προφίλ βαθμολογίας καθορίζει μια συνάρτηση με μια παράμετρο που ονομάζεται "mylocation" την επιλογή συμβολοσειρά ερωτήματος θα ήταν `&scoringParameter=mylocation--122.2,44.8`. Την πρώτη παύλα διαχωρίζει το όνομα από τη λίστα τιμών, ενώ η δεύτερη παύλα αποτελεί μέρος του την πρώτη τιμή (γεωγραφικού σε αυτό το παράδειγμα).
- Για τις παραμέτρους του βαθμολογίας, όπως για την ετικέτα ενίσχυση που μπορεί να περιέχει κόμματα, να διαφυγής οποιαδήποτε όπως τιμών σε μια λίστα χρησιμοποιώντας μονά εισαγωγικά. Εάν οι τιμές τον εαυτό τους περιέχουν μονά εισαγωγικά, μπορείτε να τους διαφυγής επιλέγοντας διπλασιασμό.
  - Για παράδειγμα, εάν έχετε μια ετικέτα ενίσχυση της παραμέτρου που ονομάζεται "mytag" και θέλετε να ενισχύσει στην ετικέτα τιμές "Hello, O'Brien" και "Κοίτα", το ερώτημα θα ήταν συμβολοσειρά `&scoringParameter=mytag-'Hello, O''Brien',Smith`. Σημειώστε ότι οι προσφορές μόνο είναι απαραίτητη για τιμές που περιέχουν κόμματα.

> [AZURE.NOTE] Κατά την κλήση **αναζήτησης** με χρήση ΔΗΜΟΣΊΕΥΣΗ, αυτή η παράμετρος ονομάζεται `scoringParameters` αντί για `scoringParameter`. Επίσης, καθορίζετε ως ένας πίνακας JSON συμβολοσειρών όπου κάθε συμβολοσειρά είναι μια ξεχωριστή `name-values` ζεύγος.

`minimumCoverage`(προαιρετικά, η προεπιλεγμένη τιμή 100)-έναν αριθμό μεταξύ 0 και 100 που υποδεικνύει ότι το ποσοστό της το ευρετήριο που πρέπει να καλύπτονται από ένα ερώτημα αναζήτησης για το ερώτημα να αναφέρονται ως επιτυχίας. Από προεπιλογή, ολόκληρο το ευρετήριο πρέπει να είναι διαθέσιμες ή `Search` θα επιστρέψει ο κωδικός κατάστασης HTTP 503. Εάν ορίσετε `minimumCoverage` και `Search` με επιτυχία, θα επιστρέψετε HTTP 200 και θα περιλαμβάνουν μια `@search.coverage` τιμή στην απάντηση που υποδεικνύει ότι το ποσοστό του ευρετηρίου που περιλαμβάνεται στο ερώτημα.

> [AZURE.NOTE] Ρύθμιση αυτής της παραμέτρου σε μια τιμή μικρότερη από 100 μπορεί να είναι χρήσιμο για την αναζήτηση διαθεσιμότητας ακόμη και για τις υπηρεσίες με μία μόνο αντιγράφου. Ωστόσο, όχι με όλα τα έγγραφα που συμφωνούν είναι εγγυημένη παρουσίαση στα αποτελέσματα αναζήτησης. Εάν είναι σημαντικότερο στην εφαρμογή σας από διαθεσιμότητα ανάκλησης αναζήτησης και, στη συνέχεια, είναι καλύτερα να αφήσετε `minimumCoverage` στην προεπιλεγμένη τιμή 100.

`api-version=[string]`(απαιτείται). Η έκδοση προεπισκόπησης είναι `api-version=2015-02-28-Preview`. Για λεπτομέρειες και εναλλακτικές εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

Σημείωση: για αυτήν τη λειτουργία, το `api-version` έχει καθοριστεί ως μιας παραμέτρου ερωτήματος στη διεύθυνση URL ανεξάρτητα από το αν καλέσετε **αναζήτησης** με ΛΉΨΗ ή ΔΗΜΟΣΊΕΥΣΗ.

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης.

- `api-key`: Το `api-key` χρησιμοποιείται για τον έλεγχο ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικό για να σας διεύθυνση URL της υπηρεσίας. Η αίτηση **αναζήτησης** να καθορίσετε ένα πλήκτρο διαχείρισης ή πλήκτρο ερωτήματος για `api-key`.

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

**Σώμα αίτησης**

Για ΛΉΨΗ: καμία.

Για την ΚΑΤΑΧΏΡΗΣΗ:

    {
      "count": true | false (default),
      "facets": [ "facet_expression_1", "facet_expression_2", ... ],
      "filter": "odata_filter_expression",
      "highlight": "highlight_field_1, highlight_field_2, ...",
      "highlightPreTag": "pre_tag",
      "highlightPostTag": "post_tag",
      "minimumCoverage": # (% of index that must be covered to declare query successful; default 100),
      "moreLikeThis": "document_key" (mutually exclusive with "search" parameter),
      "orderby": "orderby_expression",
      "scoringParameters": [ "scoring_parameter_1", "scoring_parameter_2", ... ],
      "scoringProfile": "scoring_profile_name",
      "search": "simple_query_expression",
      "searchFields": "field_name_1, field_name_2, ...",
      "searchMode": "any" (default) | "all",
      "select": "field_name_1, field_name_2, ...",
      "skip": # (default 0),
      "top": #
    }

**Συνέχιση των απαντήσεων μερικής αναζήτησης**

Μερικές φορές Azure αναζήτησης δεν είναι δυνατό να επιστρέφουν όλα τα αποτελέσματα που ζητήθηκαν σε μια μεμονωμένη απόκριση αναζήτησης. Αυτό μπορεί να συμβαίνει για διάφορους λόγους, όπως όταν το ερώτημα ζητάει πάρα πολλά έγγραφα, καθορίζοντας δεν `$top` ή να καθορίσετε μια τιμή για `$top` που είναι πολύ μεγάλο. Σε αυτές τις περιπτώσεις, θα περιλαμβάνει Azure αναζήτησης του `@odata.nextLink` σχόλιο στο σώμα απόκρισης, και, επίσης, `@search.nextPageParameters` εάν ήταν μια αίτηση POST. Μπορείτε να χρησιμοποιήσετε τις τιμές από αυτά τα σχόλια διατύπωση άλλο αιτήματος αναζήτησης για να μεταβείτε στο επόμενο τμήμα της απόκρισης αναζήτησης. Αυτή η διαδικασία ονομάζεται ***συνέχισης*** της αρχικής αίτησης αναζήτησης και τα σχόλια ονομάζονται γενικά ***διακριτικά συνέχισης***. Δείτε [το παρακάτω παράδειγμα](#SearchResponse) για λεπτομέρειες σχετικά με τη σύνταξη από αυτά τα σχόλια και όπου εμφανίζονται στο κύριο σώμα του απόκρισης. 

Τους λόγους γιατί Azure αναζήτησης μπορεί να επιστρέψει διακριτικά συνέχισης είναι συγκεκριμένης υλοποίησης και μπορεί να αλλάξει. Προγράμματα-πελάτες του ισχυρό πάντα πρέπει να είστε έτοιμοι να χειρίζεται τις περιπτώσεις όπου επιστρέφονται λιγότερα έγγραφα από όσο αναμενόταν και ενός διακριτικού συνέχισης περιλαμβάνεται για να συνεχίσετε την ανάκτηση εγγράφων. Επίσης, σημειώστε ότι πρέπει να χρησιμοποιήσετε την ίδια μέθοδο HTTP ως την αρχική αίτηση για να συνεχίσετε. Για παράδειγμα, εάν στείλει ένα αίτημα GET, τα αιτήματα συνέχισης στέλνετε πρέπει επίσης να χρησιμοποιήσετε ΓΡΉΓΟΡΑ (και με τον ίδιο τρόπο για ΔΗΜΟΣΊΕΥΣΗ).

<a name="LookupAPI"></a>
## <a name="lookup-document"></a>Αναζήτηση εγγράφου

Η λειτουργία **Αναζήτησης εγγράφου** ανακτά εγγράφου από την αναζήτηση Azure. Αυτό είναι χρήσιμο όταν ένας χρήστης κάνει κλικ σε ένα αποτέλεσμα αναζήτησης συγκεκριμένες και θέλετε να αναζητήσετε συγκεκριμένες λεπτομέρειες σχετικά με αυτό το έγγραφο.

    GET https://[service name].search.windows.net/indexes/[index name]/docs/[key]?[query parameters]
    api-key: [admin or query key]

**Αίτηση**

HTTPS είναι απαραίτητη για αιτήσεις εξυπηρέτησης. Η πρόσκληση σε **Έγγραφο αναζήτησης** μπορεί να έχει ως εξής.

    GET /indexes/[index name]/docs/key?[query parameters]

Εναλλακτικά, μπορείτε να χρησιμοποιήσετε την παραδοσιακή σύνταξη OData για την αναζήτηση κλειδιού:

    GET /indexes('[index name]')/docs('[key]')?[query parameters]

Η αίτηση URI περιλαμβάνει ένα [όνομα ευρετηρίου] και [κλειδιού], που καθορίζει ποιο έγγραφο για να ανακτήσετε από ποια ευρετήριο. Μπορείτε μόνο να λάβετε ένα έγγραφο ταυτόχρονα. Χρησιμοποιήστε την **Αναζήτηση** για πολλαπλά έγγραφα σε μία μόνο αίτηση.

**Παράμετροι ερωτήματος**

`$select=[string]`(προαιρετικό) - μια λίστα πεδίων διαχωρισμένες με κόμματα, για να ανακτήσετε. Εάν δεν έχει καθοριστεί ή δώστε `*`, όλα τα πεδία που έχει επισημανθεί ως ανακτήσιμη στο σχήμα περιλαμβάνονται στην προβολή.

`api-version=[string]`(απαιτείται). Η έκδοση προεπισκόπησης είναι `api-version=2015-02-28-Preview`. Για λεπτομέρειες και εναλλακτικές εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

Σημείωση: για αυτήν τη λειτουργία, το `api-version` έχει καθοριστεί ως μιας παραμέτρου ερωτήματος.

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης.

- `api-key`: Το `api-key` χρησιμοποιείται για τον έλεγχο ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικό για να σας διεύθυνση URL της υπηρεσίας. Η πρόσκληση σε **Έγγραφο αναζήτησης** να καθορίσετε ένα πλήκτρο διαχείρισης ή πλήκτρο ερωτήματος για `api-key`.

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

**Σώμα αίτησης**

Κανένα.

**Απόκριση**

Ο κωδικός κατάστασης: 200 OK επιστρέφεται για μια επιτυχημένη απόκριση.

    {
      field_name: field_value (fields matching the default or specified projection)
    }

**Παράδειγμα**

Αναζήτηση το έγγραφο που έχει το κλειδί "2"

    GET /indexes/hotels/docs/2?api-version=2015-02-28-Preview

Το έγγραφο που έχει '3' χρησιμοποιώντας τη σύνταξη OData αριθμό-κλειδί αναζήτησης:

    GET /indexes('hotels')/docs('3')?api-version=2015-02-28-Preview

<a name="CountDocs"></a>
## <a name="count-documents"></a>Καταμέτρηση εγγράφων

Η λειτουργία **Έγγραφα Count** ανακτά μια καταμέτρηση του αριθμού των εγγράφων στο ευρετήριο αναζήτησης. Το `$count` σύνταξη αποτελεί μέρος του πρωτοκόλλου OData.

    GET https://[service name].search.windows.net/indexes/[index name]/docs/$count?api-version=[api-version]
    Accept: text/plain
    api-key: [admin or query key]

**Αίτηση**

HTTPS είναι απαραίτητη για αιτήσεις εξυπηρέτησης. Η πρόσκληση σε **Έγγραφα του πλήθους** είναι δυνατό να δημιουργηθεί χρησιμοποιώντας τη μέθοδο GET.

Το [όνομα ευρετηρίου] στην πρόσκληση σε URI σας ενημερώνει για την υπηρεσία για να επιστρέψει το πλήθος όλων των στοιχείων στη συλλογή εγγράφων από το καθορισμένο ευρετήριο.

`api-version=[string]`(απαιτείται). Η έκδοση προεπισκόπησης είναι `api-version=2015-02-28-Preview`. Για λεπτομέρειες και εναλλακτικές εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης.

- `Accept`: Αυτή η τιμή πρέπει να οριστεί σε `text/plain`.
- `api-key`: Το `api-key` χρησιμοποιείται για τον έλεγχο ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικό για να σας διεύθυνση URL της υπηρεσίας. Να καθορίσετε την αίτηση **Έγγραφα Count** είτε ένα κλειδί διαχειριστή ή το κλειδί ερωτήματος για `api-key`.

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

**Σώμα αίτησης**

Κανένα.

Στο σώμα απόκριση περιέχει την τιμή count ως ακέραιο αριθμό με μορφή απλού κειμένου.

<a name="Suggestions"></a>
## <a name="suggestions"></a>Προτάσεις

Η λειτουργία **προτάσεων** ανακτά προτάσεις που βασίζεται σε μερική αναζήτηση εισαγωγής. Χρησιμοποιείται συνήθως στα πλαίσια αναζήτησης για την παροχή πρόβλεψης προτάσεις, καθώς οι χρήστες εισαγάγετε όρους αναζήτησης.

Πρόταση αιτήσεις στόχος που υποδεικνύει την ύπαρξη έγγραφα προορισμού, ώστε να μπορεί να επαναληφθεί το προτεινόμενο κείμενο, εάν πολλά έγγραφα υποψηφίου ταιριάζουν με την ίδια αναζήτηση εισαγωγής. Μπορείτε να χρησιμοποιήσετε `$select` για να ανακτήσετε άλλα πεδία εγγράφου (συμπεριλαμβανομένου του πλήκτρου εγγράφου), ώστε να μπορείτε να καταλάβετε το έγγραφο που θα το αρχείο προέλευσης για κάθε πρότασης.

Μια λειτουργία **προτάσεων** εκδίδεται ως μια αίτηση GET ή ΔΗΜΟΣΊΕΥΣΗ.

    GET https://[service name].search.windows.net/indexes/[index name]/docs/suggest?[query parameters]
    api-key: [admin or query key]

    POST https://[service name].search.windows.net/indexes/[index name]/docs/suggest?api-version=[api-version]
    Content-Type: application/json
    api-key: [admin or query key]

**Πότε να χρησιμοποιήσετε ΔΗΜΟΣΊΕΥΣΗ αντί για ΛΉΨΗ**

Όταν χρησιμοποιείτε HTTP GET για να καλέσετε τις **προτάσεις** API, πρέπει να έχετε υπόψη ότι το μήκος της διεύθυνσης URL της αίτησης δεν πρέπει να υπερβαίνει 8 KB. Αυτό είναι αρκετά συνήθως για τις περισσότερες εφαρμογές. Ωστόσο, ορισμένες εφαρμογές παράγει πολύ μεγάλα ερωτήματα συγκεκριμένα OData παραστάσεις φίλτρου. Για αυτές τις εφαρμογές, χρήση HTTP POST είναι καλύτερη επιλογή επειδή επιτρέπει μεγαλύτερο φίλτρα από ΛΉΨΗ. Με την ΚΑΤΑΧΏΡΗΣΗ, τον αριθμό των όρων σε ένα φίλτρο είναι ο περιορισμός παράγοντας, όχι το μέγεθος της συμβολοσειράς ανεπεξέργαστα φίλτρου επειδή το όριο μεγέθους αίτηση για ΔΗΜΟΣΊΕΥΣΗ είναι περίπου 16 MB.

> [AZURE.NOTE] Παρόλο που το όριο μεγέθους αίτηση ΔΗΜΟΣΊΕΥΣΗ είναι πολύ μεγάλο, δεν μπορεί να είναι αυθαίρετα σύνθετες παραστάσεις φίλτρου. Για περισσότερες πληροφορίες σχετικά με τους περιορισμούς πολυπλοκότητα φίλτρου, ανατρέξτε στο θέμα [σύνταξη παραστάσεων OData](https://msdn.microsoft.com/library/dn798921.aspx) .

**Αίτηση**

HTTPS είναι απαραίτητη για αιτήσεις εξυπηρέτησης. Η αίτηση **προτάσεις** είναι δυνατό να δημιουργηθεί χρησιμοποιώντας τις μεθόδους GET ή ΔΗΜΟΣΊΕΥΣΗ.

Η αίτηση URI Καθορίζει το όνομα του ευρετηρίου σε ερώτημα. Παράμετροι, όπως τον όρο αναζήτησης μερικώς εισαγωγής, έχουν καθοριστεί στη συμβολοσειρά ερώτημα στην περίπτωση αιτήσεις ΛΉΨΗΣ και, στην πρόσκληση σε προσκλήσεις σώμα στην περίπτωση ΚΑΤΑΧΏΡΗΣΗΣ.

Ως βέλτιστη πρακτική κατά τη δημιουργία αιτήσεις ΛΉΨΗΣ, μην ξεχάσετε να [κωδικοποίηση URL](https://msdn.microsoft.com/library/system.uri.escapedatastring.aspx) συγκεκριμένο ερώτημα παραμέτρων, όταν καλείτε το REST API απευθείας. Για **προτάσεις** λειτουργίες, περιλαμβάνει:

- `$filter`
- `highlightPreTag`
- `highlightPostTag`
- `search`

Κωδικοποίηση URL συνιστάται μόνο για τις παραπάνω παραμέτρους ερωτήματος. Εάν που κατά λάθος κωδικοποίηση URL τη συμβολοσειρά ολόκληρου του ερωτήματος (όλα τα στοιχεία μετά το;), θα διακόψει αιτήσεις.

Επίσης, κωδικοποίηση URL απαιτείται μόνο όταν καλείτε το REST API απευθείας με ΛΉΨΗ. Δεν υπάρχει κωδικοποίηση URL είναι απαραίτητο κατά την κλήση χρησιμοποιώντας ΔΗΜΟΣΊΕΥΣΗ **προτάσεις** ή όταν χρησιμοποιείτε το [.NET βιβλιοθήκη προγράμματος-πελάτη](https://msdn.microsoft.com/library/dn951165.aspx), η οποία χειρίζεται κωδικοποίηση URL για εσάς.

**Παράμετροι ερωτήματος**

**Προτάσεις** αποδέχεται πολλούς παραμέτρους που παρέχουν κριτήρια ερωτήματος και επίσης να καθορίσετε τη συμπεριφορά αναζήτησης. Μπορείτε να παράσχετε αυτές τις παραμέτρους στη διεύθυνση URL κατά την κλήση **προτάσεις** μέσω ΛΉΨΗ και ως JSON ιδιότητες στο σώμα της αίτησης κατά την κλήση **προτάσεις** μέσω ΔΗΜΟΣΊΕΥΣΗ συμβολοσειρά ερωτήματος. Η σύνταξη για ορισμένες παράμετροι είναι λίγο διαφορετικά μεταξύ ΛΉΨΗ και ΔΗΜΟΣΊΕΥΣΗ. Οι διαφορές αυτές επισημαίνονται ανάλογα με την περίπτωση παρακάτω:

`search=[string]`-το κείμενο αναζήτησης για να χρησιμοποιήσετε για να προτείνει ερωτήματα. Πρέπει να είναι τουλάχιστον 1 χαρακτήρα και λιγότερα από 100 χαρακτήρες.

`highlightPreTag=[string]`(προαιρετικό) - συμβολοσειράς ετικέτας που τοποθετεί μπροστά από για να πραγματοποιήσετε αναζήτηση επισκέψεων. Πρέπει να οριστεί με `highlightPostTag`.

> [AZURE.NOTE] Όταν καλέσετε **προτάσεις** χρησιμοποιώντας GET, δεσμευμένο χαρακτήρες στη διεύθυνση URL πρέπει να είναι τοις εκατό κωδικοποιημένη (για παράδειγμα, αντί για το # % 23).

`highlightPostTag=[string]`(προαιρετικό) - συμβολοσειράς ετικέτας που προσαρτά για να πραγματοποιήσετε αναζήτηση επισκέψεων. Πρέπει να οριστεί με `highlightPreTag`.

> [AZURE.NOTE] Όταν καλέσετε **προτάσεις** χρησιμοποιώντας GET, δεσμευμένο χαρακτήρες στη διεύθυνση URL πρέπει να είναι τοις εκατό κωδικοποιημένη (για παράδειγμα, αντί για το # % 23).

`suggesterName=[string]`-στο όνομα του το suggester όπως καθορίζεται στα το `suggesters` συλλογής που είναι μέρος τον ορισμό του ευρετηρίου. A `suggester` προσδιορίζει ποια πεδία ανιχνεύονται για όρους προτεινόμενες ερωτήματος. Για λεπτομέρειες, ανατρέξτε στο θέμα [Suggesters](#Suggesters) .

`fuzzy=[boolean]`(προαιρετικά, προεπιλεγμένη = false)-Όταν οριστεί στην τιμή true αυτό το API θα βρείτε προτάσεις ακόμα και αν υπάρχει ένας χαρακτήρας λείπουν ή έχουν αντικατασταθεί το κείμενο αναζήτησης. Ενώ αυτή παρέχει μια καλύτερη εμπειρία σε ορισμένα σενάρια εμφανιστεί σε μια απόδοση κόστους όπως αναζητήσεις θολό προτάσεων είναι πιο αργά και κατανάλωση περισσότερους πόρους.

`searchFields=[string]`(προαιρετικό) - τη λίστα των ονομάτων πεδίων διαχωρισμένες με κόμματα, για να αναζητήσετε το κείμενο αναζήτησης που καθορίσατε. Τα πεδία προορισμού πρέπει να είναι ενεργοποιημένη για προτάσεις.

`$top=#`(προαιρετικά, προεπιλεγμένη = 5)-τον αριθμό των προτάσεων για να ανακτήσετε. Πρέπει να είναι ένας αριθμός μεταξύ 1 και του 100.

> [AZURE.NOTE] Κατά την κλήση χρησιμοποιώντας ΔΗΜΟΣΊΕΥΣΗ **προτάσεις** , αυτή η παράμετρος ονομάζεται `top` αντί για `$top`.

`$filter=[string]`(προαιρετικό) - μια παράσταση που φιλτράρει τα έγγραφα που θεωρούνται για προτάσεις.

> [AZURE.NOTE] Κατά την κλήση χρησιμοποιώντας ΔΗΜΟΣΊΕΥΣΗ **προτάσεις** , αυτή η παράμετρος ονομάζεται `filter` αντί για `$filter`.

`$orderby=[string]`(προαιρετικό) - μια λίστα από παραστάσεις διαχωρισμένες με κόμματα, για να ταξινομήσετε τα αποτελέσματα από. Κάθε παράσταση μπορεί να είναι είτε ένα όνομα πεδίου ή μια κλήση για να το `geo.distance()` συνάρτηση. Κάθε παράσταση μπορεί να ακολουθείται από `asc` για να υποδεικνύεται αύξουσα, και `desc` για να υποδείξετε φθίνουσα σειρά. Η προεπιλογή είναι αύξουσα σειρά. Υπάρχει ένα όριο 32 όρων για `$orderby`.

> [AZURE.NOTE] Κατά την κλήση χρησιμοποιώντας ΔΗΜΟΣΊΕΥΣΗ **προτάσεις** , αυτή η παράμετρος ονομάζεται `orderby` αντί για `$orderby`.

`$select=[string]`(προαιρετικό) - μια λίστα πεδίων διαχωρισμένες με κόμματα, για να ανακτήσετε. Εάν δεν καθορίζεται, επιστρέφεται μόνο το κλειδί και προτάσεων κείμενο του εγγράφου. Μπορείτε να ζητήσετε ρητά όλα τα πεδία, ορίζοντας αυτή η παράμετρος σε `*`.

> [AZURE.NOTE] Κατά την κλήση χρησιμοποιώντας ΔΗΜΟΣΊΕΥΣΗ **προτάσεις** , αυτή η παράμετρος ονομάζεται `select` αντί για `$select`.

`minimumCoverage`(προαιρετικά, η προεπιλεγμένη τιμή είναι 80)-έναν αριθμό μεταξύ 0 και 100 που υποδεικνύει ότι το ποσοστό της το ευρετήριο που πρέπει να καλύπτονται από ένα ερώτημα προτάσεις για το ερώτημα να αναφέρονται ως επιτυχίας. Από προεπιλογή, πρέπει να είναι διαθέσιμα τουλάχιστον 80% του ευρετηρίου ή `Suggest` θα επιστρέψει ο κωδικός κατάστασης HTTP 503. Εάν ορίσετε `minimumCoverage` και `Suggest` με επιτυχία, θα επιστρέψετε HTTP 200 και θα περιλαμβάνουν μια `@search.coverage` τιμή στην απάντηση που υποδεικνύει ότι το ποσοστό του ευρετηρίου που περιλαμβάνεται στο ερώτημα.

> [AZURE.NOTE] Ρύθμιση αυτής της παραμέτρου σε μια τιμή μικρότερη από 100 μπορεί να είναι χρήσιμο για την αναζήτηση διαθεσιμότητας ακόμη και για τις υπηρεσίες με μία μόνο αντιγράφου. Ωστόσο, όχι σε όλες τις προτάσεις που ταιριάζουν είναι εγγυημένη παρουσίαση στα αποτελέσματα. Εάν η ανάκληση είναι σημαντικότερο στην εφαρμογή σας από διαθεσιμότητας και, στη συνέχεια, είναι καλύτερα να μην κάτω `minimumCoverage` κάτω από την προεπιλεγμένη τιμή 80.

`api-version=[string]`(απαιτείται). Η έκδοση προεπισκόπησης είναι `api-version=2015-02-28-Preview`. Για λεπτομέρειες και εναλλακτικές εκδόσεις, ανατρέξτε στο θέμα [Διαχείριση εκδόσεων υπηρεσία αναζήτησης](http://msdn.microsoft.com/library/azure/dn864560.aspx) .

Σημείωση: για αυτήν τη λειτουργία, το `api-version` έχει καθοριστεί ως μιας παραμέτρου ερωτήματος στη διεύθυνση URL ανεξάρτητα από το αν καλέσετε **προτάσεις** με ΛΉΨΗ ή ΔΗΜΟΣΊΕΥΣΗ.

**Κεφαλίδες αίτησης**

Η παρακάτω λίστα περιγράφει τις κεφαλίδες απαραίτητων και προαιρετικών αίτησης

- `api-key`: Το `api-key` χρησιμοποιείται για τον έλεγχο ταυτότητας την αίτηση για την υπηρεσία αναζήτησης. Είναι μια τιμή συμβολοσειράς, μοναδικό για να σας διεύθυνση URL της υπηρεσίας. Να καθορίσετε την αίτηση **προτάσεις** είτε ένα κλειδί διαχειριστή ή το κλειδί ερωτήματος ως το `api-key`.

Θα πρέπει επίσης το όνομα της υπηρεσίας για να δημιουργήσετε τη διεύθυνση URL της αίτησης. Μπορείτε να λάβετε το όνομα της υπηρεσίας και `api-key` από τον πίνακα εργαλείων σας υπηρεσίας στην πύλη του Azure. Ανατρέξτε στο θέμα [Δημιουργία μιας υπηρεσίας αναζήτησης Azure στην πύλη](search-create-service-portal.md) για περιήγηση σελίδας Βοήθειας.

**Σώμα αίτησης**

Για ΛΉΨΗ: καμία.

Για την ΚΑΤΑΧΏΡΗΣΗ:

    {
      "filter": "odata_filter_expression",
      "fuzzy": true | false (default),
      "highlightPreTag": "pre_tag",
      "highlightPostTag": "post_tag",
      "minimumCoverage": # (% of index that must be covered to declare query successful; default 80),
      "orderby": "orderby_expression",
      "search": "partial_search_input",
      "searchFields": "field_name_1, field_name_2, ...",
      "select": "field_name_1, field_name_2, ...",
      "suggesterName": "suggester_name",
      "top": # (default 5)
    }

**Απόκριση**

Ο κωδικός κατάστασης: 200 OK επιστρέφεται για μια επιτυχημένη απόκριση.

    {
      "@search.coverage": # (if minimumCoverage was provided in the query),
      "value": [
        {
          "@search.text": "...",
          "<key field>": "..."
        },
        ...
      ]
    }

      "@search.coverage": # (if minimumCoverage was provided in the query),
      "value": [
        {
          "@search.text": "...",
          "<key field>": "..."
          <other projected data fields>
        },
        ...
      ]
    }
